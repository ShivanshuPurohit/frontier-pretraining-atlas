14. SOTA dataloading and I/O at 100k GPUs: the bottleneck nobody photographs

There is a photogenic version of a 100,000-GPU run — the rack rows, the NVLink spine, the loss curve bending. The dataloader appears in none of it, which is exactly why it kills runs. At this scale the loader stops being a per-host detail and becomes a distributed storage system with a hard real-time SLA: every microbatch boundary, on every one of ~12,500 hosts (8 GPU/host across ~1,389 NVL72 racks), the accelerators must already own their tokens or they idle. A 100,000-GB200 cluster at ~0.45 EFLOP/s dense FP8 peak and a realistic 35–40% large-MoE FP8 MFU (NVIDIA GB200 NVL72; sparse-vs-dense footnote) is burning on the order of $30–60/s in depreciation. A 200ms host stall per step, repeated, is a multi-percent tax on the most expensive compute humanity has ever pointed at a corpus. The loader is also the single component most likely to be non-deterministic on restart, and a 10T-token MoE run will restart dozens of times. This chapter is the punch-list for making the I/O layer invisible — which is the only acceptable state for it to be in.

14.1 The five hard requirements

A frontier loader at 100k hosts must satisfy all five of these simultaneously; most loaders satisfy three or four, and the missing one is where the run bleeds.

  1. Deterministic. The exact token stream is a pure function of (seed, step, world-topology-invariant). Two runs from the same checkpoint must produce bit-identical batches. This is not pedantry: it is the only way to debug a loss spike, reproduce a NaN, or trust an ablation.
  2. Checkpointable in O(small). Resume state must be tiny and writable on the critical path without stalling the accelerators. Per-host iterator blobs that scale with process_count are a liability — at 12,500 hosts they become 12,500 small-object reads on every restart, the exact access pattern that triggers object-store throttling.
  3. Exactly-sharded. Every token is seen exactly once per epoch, partitioned disjointly across data-parallel replicas with no overlap and no gaps. Silent over- or under-sampling at the shard boundary corrupts the data-constrained scaling budget you cannot afford to corrupt.
  4. No host stalls. Shard decode, detokenize-free collation, packing, and host→device transfer must be fully overlapped with compute and never block. This means colocated, multi-threaded, prefetched loaders with no CPU-heavy stage on the hot path.
  5. Fault-tolerant and elastic. The stream must survive preemption, host failure, and — the hard one — a change in world size. When you lose a rack and reshard from N to N′ hosts, the loader must resume the same logical stream, not silently re-shuffle or re-emit.

The MaxText input pipeline (the user's actual codebase, MaxText) routes dataset_type to one of grain | hf | olmo_grain | tfds | c4_mlperf | synthetic in input_pipeline_interface.py, and only the Grain-family paths (grain, olmo_grain) plus, in the broader field, MosaicML StreamingDataset and Megatron-Energon credibly hit all five. The rest fail requirement 1, 3, or 5.

14.2 The bandwidth is trivial; the request rate is lethal

Start with the bytes. Pre-tokenized IDs at a DeepSeek-class >100k vocab need uint32 (4 B/token); uint16 bit-packing (2 B/token) is off the table without sub-vocab remapping. A 10T-token corpus is therefore 40 TB on disk as raw IDs. A 200B-active MoE on 100k GB200 plausibly consumes O(10^7–10^8) tokens/s wall-clock. The sustained read bandwidth this implies:

Throughput Bytes/token Sustained read
2×10^7 tok/s 4 B 80 GB/s
1×10^8 tok/s 4 B 400 GB/s

A single GCS or S3 bucket sustains multi-hundred-GB/s aggregate without complaint. Bandwidth is a non-problem. The wall is request rate. GCS flat-namespace buckets start at ~5,000 object reads/s and ~1,000 writes/s per bucket, auto-scale no faster than doubling every ~20 minutes, and top out near 80k reads/s only with good key distribution; hierarchical-namespace (HNS) buckets give up to 8× higher initial QPS (GCS request-rate guidelines). S3 sustains 5,500 GET/HEAD and 3,500 PUT per partitioned prefix, returning 503 SlowDown while it splits partitions in the background (S3 performance guidelines).

Now do the arithmetic that should keep you up at night. 12,500 hosts issuing even 10 reads/s each = 125,000 req/s — 25× a flat GCS bucket's cold-start ceiling, 23× a single S3 prefix. At job launch and after every preemption restart, 12,500 hosts cold-start the same keyspace simultaneously. That is a textbook thundering herd, and it manifests as a 503/429 storm precisely at the moment you most want the run back up. The bandwidth math says "easy"; the request math says "you will be throttled to a crawl unless the access pattern is engineered." The entire art of dataloading at this scale is converting many small requests into few large sequential ones, and spreading them across the keyspace and across time.

The structural mitigations, in priority order:

  • Large shards. ArrayRecord or WebDataset shards of 256 MB–1 GB so one GET amortizes thousands of samples. This is the single biggest lever on request count.
  • Prefix fan-out. Hash shard names across thousands of prefixes so S3/GCS partition the keyspace and your QPS ceiling multiplies by the prefix count.
  • HNS buckets for both data and checkpoints — the 8× initial-QPS multiplier is free if you provision the bucket type correctly up front.
  • Staggered exponential-backoff ramp so 12,500 hosts don't cold-start the same prefix in the same second. Jittered startup is mandatory, not optional.
  • A read-amplification-reducing loader that holds shards open (mmap or HTTP range-read) rather than re-GETting per sample.

WebDataset's purely sequential tar reads are 3–10× faster than random access on local disk and are object-store-friendly because they fold N small GETs into one streamed GET. ArrayRecord trades pure-sequential for true random access via its Riegeli-backed per-record index (Grain + ArrayRecord) — which is what buys you a real global shuffle. That tradeoff (sequential-tar throughput vs random-access shuffle quality) is the fault line dividing the loader matrix below.

Grain + ArrayRecord — the MaxText default, and the right default

Grain's IndexSampler is the only widely-deployed training loader that does a true global shuffle over a non-resident dataset: it generates a permuted index stream on the fly and random-accesses ArrayRecord records by key, because Riegeli embeds a per-record position index (Grain + ArrayRecord). Determinism is a pure function of (seed, shard_options, epoch). The MaxText wiring in grain_data_processing.py is the canonical shape: MapDataset.source(ArrayRecordDataSource) → .shuffle(seed) → .repeat(num_epoch) → host-strided sharding → to_iter_dataset(ReadOptions(num_threads, prefetch_buffer_size)), with grain_num_threads and grain_prefetch_buffer_size as the host-stall-avoidance knobs.

Two non-obvious design choices in that file are load-bearing at frontier scale:

  1. Mixtures use grain.IterDataset.mix(weights), not MapDataset.mix. MaxText does this specifically so each mixture component checkpoints independently, which means you can re-weight the curriculum after a restart without corrupting per-component position state (the user's actual codebase, MaxText, grain_data_processing.py). For a long curriculum that shifts toward math/code/reasoning after LR decay, this is the property that lets you change the mix at step 800k without a fresh data pipeline.
  2. The elastic path. grain.experimental.ElasticIterator (enabled via grain_use_elastic_iterator) does sharding and batching internally from a single global_batch_size, and its checkpoint state is a single global scalar. MaxText exploits this in multihost_dataloading.py: under elastic mode only process 0 writes process_0.json, and every process reads that same file on restore — so the iterator survives a change in jax.process_count() (the user's actual codebase, MaxText). This is the cleanest elastic-resize-survival design among the JAX loaders, and it directly kills the requirement-5 failure mode.

Note the trap in the non-elastic Grain path: it writes process_{i}-of-{n}.json per host. That blob is correct, but it is N files written and N files re-read on restart — 12,500 small-object operations against the checkpoint bucket on every resume, exactly the request pattern §14.2 says to avoid. At 100k hosts you want the elastic path or the stateless sampler below, not per-host JSON.

MaxText's OLMo Grain loader — the stateless-resume blueprint

src/maxtext/input_pipeline/olmo_data_grain.py is the most instructive in-repo artifact for your scale, because it eliminates iterator-state checkpointing entirely. Instead of serializing position, OlmoIndexSampler.__getitem__(i) is a pure function: it maps per-host step i to a record key via a single Fisher-Yates permutation over [0, total_instances) keyed by SHA-256("olmo-shuffle:{seed}:{epoch}") truncated to 64 bits (the user's actual codebase, MaxText — the loader explicitly uses SHA-256 "because it has no fixed points unlike a raw multiply"). The epoch is (i + initial_step) // n_per, the in-epoch position is (i + initial_step) % n_per, and resume needs zero serialized iterator state: you pass initial_step = train_step × per_host_batch at construction and the (seed, shard, absolute_step) tuple fully determines the next record (olmo_data_grain.py).

The cleverest detail is a one-line compatibility hack with deep consequences: __repr__ deliberately excludes initial_step so that Grain's repr-based sampler-equality check still accepts a resumed loader as "the same" sampler — a resumed run with a different offset must not look like a different pipeline to Grain's bookkeeping (the user's actual codebase, MaxText). This is the pattern you want at 12,500 hosts: zero per-host checkpoint state on the critical restart path, so a resume triggers no small-object read storm at all. The olmo_data_grain_resume_test.py pins this contract explicitly.

The loader also documents its own scaling defect in-code, which is gold. shuffled_global_indices materializes a full numpy rng.permutation(total) — ~5.8 GB at uint64 for the 724M-instance OLMo mix, allocated per epoch per host — and the source comment says production should swap to OLMo-core's on-disk build_and_save_global_indices memmap scheme (olmo_data_grain.py; OLMo-core data loading). At trillions of instances that allocation is fatal twice over: it overflows RAM, and OLMo-core's uint32 index format itself overflows at 4.29B instances, forcing uint64 or a block-structured index. The fix is to compute the permutation once, write global_indices.npy as a memmap, and mmap-share it across all forked Grain workers — never recompute in RAM.

The supporting OlmoNpyDataSource shows the right fork-safety discipline: per-worker LRU-cached np.memmap handles with max_open_files=64 over the 950-file mix (to stay under ulimit -n), and it copies out of the read-only memmap before returning so forked Grain workers can pickle the result without dragging the mmap object across the fork boundary (the user's actual codebase, MaxText). No shared mutable state across fork is the whole game for host-stall avoidance — a shared mmap that must be re-faulted or a lock that serializes workers is how you reintroduce stalls.

MosaicML StreamingDataset — the other production-grade elastic-deterministic loader

Streaming's elastic determinism is real but conditional: it survives a world-size change iff the global batch size stays constant and is divisible by every GPU count you run on, and num_canonical_nodes (NCN) is pinned across runs (Streaming elastic determinism). NCN partitions the sample space into buckets (default = physical node count of the first run, or 64× under py1s/py2s). The default shuffle is py1e with shuffle_block_size = max(4×10^6 / NCN, 1<<18) (floor 262,144) and shuffle_seed = 9176 (Streaming shuffling reference). The algorithm zoo and its tradeoff:

Algo Shuffle quality Download locality Disk footprint
py1e best balance good moderate
py1br good download-optimal low
py1s / py2s lower good minimal
naive true global destroys locality single-node only

The design lesson generalizes to any 100k-host loader: py1e deliberately shuffles within a block rather than globally, to preserve object-store read locality. naive global shuffle is unusable at scale precisely because it converts every sample read into a random GET against a random shard — the §14.2 catastrophe. Mid-epoch resumption replays the deterministic order without re-downloading already-seen shards. The tuning tension at 100k GPUs: NCN must be large enough that bucket count ≥ node count and gives download diversity, but shuffle_block_size shrinks as NCN grows (it is 4e6/NCN), trading shuffle quality for cache footprint. You tune the NCN × block-size product against per-node SSD cache size — and you treat both NCN and shuffle_seed as part of run identity.

Megatron-Energon + WebDataset, and the legacy indexed mmap

If the stack is PyTorch/Megatron-Core rather than JAX, Megatron-Energon is the Grain-equivalent. It layers .nv-meta (dataset.yaml, split.yaml) over WebDataset tar shards and provides savable/restorable loader state, deterministic multi-dataset blending, built-in packing/grouping/joining, and native S3/GCS/Azure streaming (Megatron-Energon features). Its sequential-tar I/O is the most object-store-throughput-friendly format in the field. The catch: it is per-rank deterministic but does not offer Streaming's elastic-determinism across arbitrary world-size changes — so on Energon you pin your topology or accept a re-shuffle on resize.

Megatron-LM's indexed .bin/.idx mmap dataset is the legacy workhorse. IndexedDataset (.bin payload + .idx metadata) and GPTDataset build three cached .npy memmaps — a document index (Do_idx, shuffled), a sample index (Sa_idx, shape [N+1,2] giving sample bounds), and a shuffle index — and blend datasets via a compiled build_blending_indices in helpers.cpp (Megatron-Core datasets). It is deterministic, fast, and exerts zero object-store request pressure once staged — but it assumes the corpus is staged to local NVMe. At 10T tokens × 4 B = 40 TB, that is feasible as a per-rack-local stage but forces an explicit copy step and breaks the streaming/elastic story. It also has a known BlendedDataset IndexError in extreme many-component blends (issue #705) that bites exactly when you have hundreds of curriculum sources. For a 100k-GB200 streaming run, indexed-mmap is the wrong default; it is the right tool only if you can fully stage the corpus rack-local and never reshape.

tf.data / HF / Ray / Daft — know their lane

tf.data (+ tf.data service) pipelines beautifully but has weak fine-grained checkpoint/resume, and TFRecord's sequential format cannot do a true global shuffle — window-shuffle only. MaxText reflects this honestly: for tfrecord/parquet it falls back to a WindowShuffleIterDataset (the user's actual codebase, MaxText). HF IterableDataset shard-and-resume is improving but is not the tool for deterministic 100k-host training. And Daft / Ray Data are offline-prep engines, not training loaders — which is precisely their correct role. The proof point is Essential-Web v1.0: 24T tokens / 23.6B documents from CommonCrawl, processed in a 7-day window with "0 crashes" on a Daft pipeline sustaining 32,000 requests/s per VM (Essential AI / Daft case study). The pattern crystallizes the whole chapter: do the heavy global shuffle, dedup, and offline pre-tokenization in Daft/Ray/Spark, emit large ArrayRecord/WebDataset shards, and keep the training loader dumb, deterministic, and stateless.

14.4 Tokenization: offline, always, at this scale

Online tokenization is malpractice at 100k hosts. It burns the exact host CPU the loader needs for shard decode and collation, and it puts a variable-latency, branchy CPU stage on the hot path — the textbook source of host stalls. HF's Rust tokenizers runs ~1 GB text / 20 s / core ≈ 12.5M tokens/s/core (HF tokenizers). A 10T-token corpus is ~40 TB of text ⇒ ~10^6 core-seconds — a few thousand cores for a few hours, embarrassingly parallel inside the same Daft/Ray job that does dedup and shuffle. Tokenize once, store uint32 IDs in ArrayRecord/.bin, and the training loader becomes a pure index → fetch → collate function with no tokenizer in sight. The marginal cost of offline pre-tokenization is rounding error against the pretrain; the cost of online tokenization is a permanent MFU tax plus a non-reproducibility risk if the tokenizer version ever drifts.

14.5 Packing, masking, and the global-ordering problem

Packing. MaxText's MoE-grade path is greedy online packing (packing/sequence_packing.py) emitting inputs_segmentation / inputs_position, consumed by the TPU splash-attention kernel (GPU: flash-attention varlen) for intra-document masking with no cross-document attention; the c4_mlperf pipeline asserts packing is on (the user's actual codebase, MaxText, input_pipeline_interface.py). Two engineering notes:

  • Best-fit-decreasing bin-packing (sort documents descending, place each in the tightest-fitting bin) minimizes pad waste versus first-fit. Do it offline so the packing itself is deterministic and checkpointable — online greedy packing is order-dependent and complicates exact resume.
  • The splash-attention divisibility constraint is real and visible in the OLMo loader: q_seq_len must be divisible by 512, so the loader emits the full length L and masks the boundary position rather than producing L-1 (the user's actual codebase, MaxText — olmo_data_grain.py comments this explicitly). Get this wrong and the kernel silently rejects or mis-shapes batches.

To mask cross-document attention or not. This is a genuine fork, not a default. The controlled study on sequence composition shows intra-document causal masking achieves lower perplexity than naive random packing or semantic grouping (arXiv:2402.13991). Yet DeepSeek-V3 trains on 14.8T tokens using document packing and deliberately does not apply cross-sample attention masking during pretraining — it applies it only during SFT — as an efficiency choice (DeepSeek-V3). The MaxText splash/flash-varlen path makes intra-doc masking cheap enough that for a 200B-active MoE I would take the perplexity win: the kernel cost is small relative to the MoE all-to-all, and the no-cross-doc-attention property removes a class of subtle long-context contamination. But the decision is workload-specific; if your packed sequences are mostly single long documents (high chunk_attn_window_size, long-context curriculum), the masking benefit shrinks and DeepSeek's choice is defensible.

True global shuffle at 10T tokens. A uniform global shuffle needs either ArrayRecord random-access (Grain's index stream) or an offline permutation materialized as an on-disk index (OLMo-core global_indices.npy). Both Streaming's py1e and Grain's index stream are bucketed approximations — they shuffle within canonical-node/shard buckets to preserve download locality — and that is the correct tradeoff, because a naive truly-global shuffle destroys object-store read locality and is unusable at this scale. Remember the integer-width trap: a uint32 index overflows at 4.29B instances; a 10T-token corpus packed to ~2k-token sequences is ~5B instances, already over the line, so plan on uint64 or block-structured indices from day one.

14.6 The punch-list: what to fix at 100k hosts, concretely

Ordered by leverage. These map directly onto MaxText/Grain knobs you can set today.

  1. Don't have 12,500 hosts read GCS. Set expansion_factor_real_data > 1. Only jax.process_count() / factor hosts load real data; the rest receive batches over the device mesh via get_process_loading_real_data + PlaceHolderDataIterator in input_pipeline_interface.py (default in base.yml is -1.0 = off; the comment states each loading host then loads per_device_batch_size × factor). This cuts object-store request fanout by the factor directly and is the single highest-leverage knob (MaxText Grain data-input guide; the user's actual codebase, MaxText). At a factor of 8 you go from 12,500 readers to ~1,560 — back inside an HNS bucket's headroom.
  2. Colocate the loader. Set colocated_python_data_input=True (the RemoteIterator path in multihost_dataloading.py) so Grain runs on the accelerator host and shard decode + collation use host CPU/RAM with NVLink/ICI-adjacent transfer instead of PCIe round-trips, eliminating a class of host stalls (the user's actual codebase, MaxText; flagged "experimental, under testing" in base.yml).
  3. Choose a stateless or single-scalar resume path. Either ElasticIterator (single global-scalar state, process_0.json only, survives process_count change) or the OLMo initial_step sampler (zero serialized state). Avoid the per-host process_{i}-of-{N}.json path — it is N writes and N reads on every restart, a small-object storm against the checkpoint bucket at 12,500 hosts.
  4. Move the shuffle permutation to memmap. Adopt OLMo-core's build_and_save_global_indices to kill the ~5.8 GB/host/epoch rng.permutation allocation the MaxText OLMo loader explicitly warns about. Compute once, uint64, mmap-share across forked workers (olmo_data_grain.py).
  5. Provision storage for the request wall: HNS buckets (8× initial QPS) + ≥256 MB shards + thousands of name-hashed prefixes + jittered exponential-backoff ramp so the 12,500-host cold start and every post-preemption restart clears the 5,000-reads/s cliff instead of triggering a 429/503 storm (GCS request-rate; S3 performance).
  6. Pin shuffle identity. Treat data_shuffle_seed (and StreamingDataset's num_canonical_nodes if you use it) as part of run identity, frozen in config, so any elastic resize stays bit-deterministic. A reshard that silently re-seeds the shuffle is an undetectable correctness bug.
  7. Pre-tokenize and pre-pack offline (best-fit-decreasing, deterministic) so the training loader is a pure deterministic index→fetch→collate with no CPU-heavy stage on the critical path — the Essential-Web/Daft pattern of decoupling heavy offline work from a dumb online loader (Essential AI / Daft).

14.7 Recommendation for the 10T / 200B-A / 100k-GB200 target

Build the offline pipeline in Daft or Ray Data: dedup, global shuffle, pre-tokenize to uint32, best-fit-decreasing pack, and emit 256 MB–1 GB ArrayRecord shards across thousands of HNS-bucket prefixes. On the training side, run Grain with the OLMo-style stateless pure-function sampler backed by an on-disk uint64 global_indices memmap — zero per-host resume state, which is the property that makes restarts free at 12,500 hosts. Drive object-store fanout down with expansion_factor_real_data ≈ 8 and colocated_python_data_input=True. Use grain.IterDataset.mix for the curriculum so you can re-weight toward math/code/reasoning after LR decay without rebuilding the pipeline. Turn on intra-document masking via splash/flash-varlen (take the perplexity win; respect the q_seq_len % 512 == 0 constraint). Freeze data_shuffle_seed and your shard layout as run identity.

The success criterion is brutal and simple: the loss curve never knows the loader exists. No step ever waits on a GET; no restart ever triggers a 503 storm; every resume reproduces the exact stream as if the run was never interrupted. The dataloader earns its invisibility — which is the only photograph of it worth taking.
