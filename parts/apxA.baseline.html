Appendix A. The scaling cheat-sheet — every formula you need, in one place

This is a dense reference, not prose. Every formula below is drawn from the JAX How To Scale Your Model book (landing) and its referenced papers. Hardware-specific constants default to GB200/Blackwell where the book gives a Blackwell number; TPU v5e/v5p values are retained where they are the only anchor, since the book's symbolic results are hardware-agnostic and you re-instantiate the constants C (peak FLOPs/s) and W (bandwidth) for your silicon.

Convention throughout: B=batch in tokens (not sequences), T=query seq len, S=key/value seq len, D=d_model, F=d_ff, L=layers, N=query heads, K=KV heads, H=head dim, G=N/K (GQA group size), V=vocab, E=experts, k=experts/token, C=accelerator peak FLOPs/s, W=bidirectional link bandwidth (bytes/s), X/Y/Z=devices on a mesh axis, M_X/M_Y=number of mesh axes a strategy uses.

A.2 The roofline (the master rule)

Quantity Formula Use it when
Compute time T_math = FLOPs / C always — the floor if perfectly compute-bound
Comms/mem time T_comms = Bytes / W always — the floor if perfectly bandwidth-bound
Runtime (overlap) T = max(T_math, T_comms) good kernels; the book's default estimate
Runtime (no overlap) T = T_math + T_comms pessimistic upper bound
Arithmetic intensity I = FLOPs / Bytes classify any op
Accelerator intensity I_acc = C / W the ridge point (240 v5e, 281 B200 HBM)
Compute-bound iff I > I_acc ⇔ T_math > T_comms the single most-used decision in the book
Achievable FLOP/s total_FLOPs / max(T_math, T_comms) for roofline plots / MFU

Rule: runtime is bounded by max(compute, comms, memory). A 20% benchmark win is worthless at a 20% roofline-efficiency cost.

A.3 Matmul FLOPs, bytes, and the critical batch

For X[B,D] · Y[D,F] → Z[B,F]:

Quantity Formula Note
FLOPs 2·B·D·F factor 2 = multiply + add
Bytes (bf16) 2BD + 2DF + 2BF load both inputs + write output
Intensity BDF/(BD+DF+BF) ≈ B when B ≪ D,F intensity ≈ token batch
Compute-bound B > C/W_hbm ≈240 (v5e), ≈281 (B200), ≈295 (H100)
Contraction FLOPs (general) 2 × (product of all input dims, batch & contract dims counted once) no factor 2 for pure elementwise
Tiled-from-VMEM intensity bm·bn/(bm+bn); threshold drops to ~10–20 VMEM/SMEM is ~22× HBM BW

Precision levers (critical batch B_crit):

Scheme B_crit (v5e) Why
bf16 weights + bf16 FLOPs 240 baseline C/W_hbm
int8/fp8 weights, bf16 FLOPs 120 half the weight bytes, same FLOPs
int8 weights + int8 FLOPs 240 both peak ops/s and intensity double
per-example weight matrix (no reuse) n/a — intensity ≈ 2 const always comms-bound, avoid in hot paths

General: B_crit = (C/W_hbm) · (bits_per_param / bits_per_activation).

Sharded 2-chip matmul (shard contracting D): compute-bound iff D/(2X) > C/W — for large models (D≈8k) this needs X<2, i.e. you are essentially always comms-bound on the contracting dim; prefer the all-reduce strategy. See Rooflines, Sharding.

A.4 Transformer FLOPs (training & inference)

Master rule (Transformers):

Training FLOPs ≈ 6 · num_params · num_tokens (2 fwd + 4 bwd per matmul). Inference ≈ 2 · num_params · num_tokens.

Per-layer FLOPs (use to find which component dominates):

Component FLOPs (training, ×6/×2 of fwd) Params Use it when
MLP (gated/SwiGLU) 18·B·T·D·F 3DF dominates while T < 8D
MLP (non-gated) 12·B·T·D·F 2DF single W_in
Attn QKVO proj 12·B·T·D·(N+K)·H 2D(N+K)H GQA shrinks K-side
Dot-product attn 12·B·T·S·N·H ≈ 12BT²NH 0 matters only at long ctx
Unembedding 6·B·T·D·V DV not per-layer
LayerNorm O(BTD) D negligible

Attention fraction (with F=4D, D=NH, N=K): ≈ T/(8D). Dot-product attention dominates training FLOPs only once T > 8D (~64K tokens at D=8k; ~37k at D=4608). Self-attn FLOPs = QKVO FLOPs at T = 2D. Causal masking halves useful attention FLOPs (needs a real kernel). ~¼ of params live in attention, ¾ in MLP. SwiGLU's 3-matrix FFN keeps params/FLOPs fixed by setting d_ff = (2/3)· the ungated width (GLU Variants).

MoE: params ×O(E), activated params/token ×k, sparsity E/k (8–64). Two all-to-alls (route + un-route) when sharded on the expert axis, each ¼ of a single-axis all-gather. MoE compute-bound batch: B > 120·E/k (DeepSeek-V3: E=256, k=8 → B>3840).

A.5 Memory: parameters, optimizer, KV cache, activations

Parameter + optimizer state per parameter:

State Bytes/param Use it when
bf16 weights 2 inference / fwd
fp32 Adam (master + m + v) 8 (=4+4+... master copy folded) → 10 total w/ bf16 params training memory ceiling
Muon (momentum only, no 2nd moment) ~4 (fp32 momentum) → ~6 total w/ bf16 params ~40% less optimizer state than Adam; the second-moment buffer is replaced by an orthogonalized update, so you drop one fp32 buffer/param

Max model under pure DP+Adam = HBM_per_device / 10 (≈9B on v5p; ≈19B on a 192 GB B200). Under Muon (~6 B/param) the same device holds ~1.6× more parameters.

KV cache (Transformers, Inference):

Quantity Formula Note
Per token (precision p) 2 · L · K · H · p 2 = keys+values; GQA uses K not N
Full cache 2 · S · L · K · H · p scales with batch — this dominates inference memory
MHA per token (int8) 2·L·N·H e.g. 512 kB/token at D=4096, L=64
GQA reduction × H/G vs MHA LLaMA-3: N=64, K=8 → 8× smaller
MLA (DeepSeek-V3) cache compressed latent d_c=512 + RoPE key d_h^R=64 only drastic further cut (DeepSeek-V3)

Standard inference: per-token KV-cache footprint = 2 · hidden · num_layers · bytes_per_param (vLLM accounting); OPT-13B FP16 = 800 kB/token, 1.6 GB at 2048 ctx (vLLM/PagedAttention). Block size 16 tokens; paging cuts the 62–80% fragmentation waste of Orca/FasterTransformer to ~0.

Activation memory (Transformers):

Scheme Bytes (bf16) FLOP cost
No remat ≈ 2 · 20 · B · T · D · L (~20 nodes/layer) 6×
Block remat (1 ckpt/layer) 2 · B · T · D · L ~8× (recompute fwd in bwd)
Big-matmuls-only (~7 nodes/layer) 2 · 7 · B · T · D · L between
Checkpointed (3 big matmuls) 2·L·B·(D + 2F) activation rarely the bound at scale

Example: B·T=4M, L=64, D=8192 → 84 TB no-remat; 4.2 TB block remat.

FlashAttention makes attention scratch O(N) not O(N²): Θ(N²d²/M) HBM accesses (M=SRAM), block sizes B_c=⌈M/4d⌉, B_r=min(⌈M/4d⌉,d) (FlashAttention). It does more GFLOPs (75.2 vs 66.6) but 9× fewer HBM bytes (4.4 vs 40.3 GB) → 5.7× faster. Rabe-Staats lazy-softmax gives O(log n) / O(√n) memory on TPU (arXiv:2112.05682).

A.6 Collective costs — bytes on the wire and lower-bound time

V = total array bytes, W = bidirectional link bandwidth, |X| = shards on the axis, num_axes = mesh axes the collective spans. All ring costs are independent of device count. (Sharding)

Collective Time (bandwidth-bound) Relative cost Use it when
AllGather V / (W · num_axes) 1 replicate a sharded array
ReduceScatter V / (W · num_axes) 1 sum + re-shard (= AG transpose)
AllReduce 2V / (W · num_axes) 2 = RS + AG; sum to replicated
AllToAll (1D ring) V / (4W) ¼ reshard / MoE routing — cheap!
AllToAll (ND mesh) V·max(axes) / (4·N·W) — N = total devices

Cost hierarchy: AllToAll (¼) < AllGather = ReduceScatter (1) < AllReduce (2). AllGather and ReduceScatter are transposes: a fwd all-gather becomes a bwd reduce-scatter.

Per-hop derivation: T_hop = 2V/(|X|·W), ~|X|/2 hops → V/W. Incomplete rings (axis without wraparound) lose bidirectionality (~2× slower; a 4-device edge axis = 3 hops).

Latency floor: T = max(T_min·|X|/2, V/W), T_min ≈ 1 µs/hop. A buffer is latency-bound below W_unidir·T_min per device = 45 kB/device on v5e. AllGather hits ~95% of peak at ~10 MB (~625 kB/device 16-way).

GPU collectives (GPUs):

Collective GPU formula Note
Intra-node AG/RS bytes·(N−1)/(N·W_egress) ≈ bytes/W_egress ≈ bytes/900e9 on B200
Intra-node AllToAll (dense) B·(N−1)/(W·N²) ≈ B/(W·N) 2× faster than TPU ring (B/4W vs B/8W)
Ragged AllToAll (k of N experts) × min(k/N, 1) MoE win
AllReduce w/o SHARP 2 × AllGather RS+AG
AllReduce w/ SHARP (in-NVSwitch) bytes / W_egress theory 2×, ~30% measured
Cross-node AG/RS (tree) bytes · max_i[(D_i−1)/(D_i·W_link,i)] H100 SuperPod ≈ 413 GB/s effective
Cross-node AllToAll B·(M−1)/(M²·W_node) ≈ B/(M·W_node), M=nodes ~50 GB/s vs 400 intra — >4× degradation

CollectiveEinsum / collective-matmul (Wang et al.) overlaps the ring with the local matmul; measured 311 µs (blocking AG) → 244 µs (overlapped) → 224 µs (unsharded baseline) on v5e-8.

A.7 Per-parallelism-strategy comms volume & the bandwidth-bound condition

For a transformer MLP block; α = C/W (the "operational intensity" of the network, ≈2550 on a single v5p ICI axis). (Training, GPUs)

Strategy Comms/layer (bytes) Compute-bound condition Bandwidth-bound when Typical limit
Data Parallel 2 AllReduce of weight grads = 8DF B/X > C/W per-dev batch < C/W needs ≥2500 tok/GPU (B200: ~5625; v5p 1-axis ~2550, 3-axis ~850)
FSDP / ZeRO-3 AG weights 4DF fwd + RS grads 8DF bwd B/X > C/W per-dev batch < C/W same as DP but shards memory; can't exceed #sequences
Tensor (Megatron) AG+RS activations 4BD fwd & bwd Y < F·W/C i.e. Y < M_Y·F/2550 Y > F·W/C ~8-way (8 on H100/v5e, ≤16); precision-independent
Pipeline adjacent-stage acts only, ≈1.5·2BD/(W·L) ~always (single-hop) bubble, not bandwidth needs zero-bubble schedule + microbatch
Expert (EP, MoE) 2 AllToAll (route/un-route); 4BD·(Z−8)/(WZ)·min(8k/Z,1) F > 8·C/W_node (multi-node) cross-node AllToAll keep within node/few nodes; prefer over TP
Context/Seq = DP for MLP ("tokens are tokens"); attn needs KV gather / ring-attn B/X > C/W same extends DP past #sequences

FSDP + TP combined (the workhorse at 100k-GPU scale):

Quantity Formula
Total comms time max(FSDP comms, TP comms) = max(12DF/(Y·W·M_X), 12BD/(X·W·M_Y))
Optimal FSDP degree X_opt = sqrt((B/F)·(M_X/M_Y)·N) (also √(2BN/F)), N=XY
Compute-bound iff B/N > α²/(M_X·M_Y·F) = 2550²/(2F) ≈ 100 tok/chip (F=32768)

This ~8× reduction (850 → ~100 tok/chip) vs pure FSDP is the reason to combine. Model parallelism that spans nodes (PP/EP/TP) divides the DP all-reduce cost, letting you stay compute-bound at smaller per-GPU batch.

Cross-pod / DCN (Training): T_comms = 8DF/(M·W_dcn), M = chips/slice; compute-bound iff per-pod batch > C/W_dcn (= 73,440 tok/pod on v5p). DeepSeek-V3 used 16-way PP × 64-way EP (8 nodes) × 2-way DP, no TP, at ~30k tok/GPU, NVLink ~160 GB/s / IB ~50 GB/s, all-to-all on 20 of 132 SMs, DualPipe hiding comms (DeepSeek-V3). 2D weight-stationary FFN sharding (Pope et al.): T_comm = 8BLE/(√n·BW), beats 1D when √n_chips > d_ff/d_model i.e. n_chips > 16 (ESTI).

A.8 Inference rooflines

Quantity Formula Use it when
Linear T_math 2BDF / C prefill / MLP
Linear T_comms (2BD+2FD+2BF)/W_hbm small batch
Compute-bound B ≥ B_crit = C/W_hbm 240 v5e, ~280 B200/H100
Attn intensity (MHA) ST/(S+T) prefill =T/2 (compute-bound >~480 tok); gen ≈1 (always BW-bound)
Step time (general) B·KV/W_hbm + max(2B·P/C, P_bytes/W_hbm) P=params; 1st term attn (always BW-bound), 2nd MLP
Min step time (B·KV + P_bytes)/W_hbm decode latency floor = load weights+KV once
Max tokens/s B·W_hbm / (B·KV + P_bytes) throughput ceiling
Gen model-parallel crossover ICI-bound when Y > F/(B·β), β = W_hbm/W_ici ≈ 8 shard further when memory-bound
2D weight-stationary T = 2BD/(X·W) + 4BF/(YZ·W), beats 1D when N > 32(F/D)(¾)² (~72 chips) large-N decode
Latency-bound collective shard buffer < 45 kB · n_shards sub-µs/shard

Prefill is essentially always compute-bound (maximize MFU). Generation MLP switches at B_crit; generation attention is never compute-bound (G small). For throughput, use the largest per-chip batch that fits. Multi-query/GQA decode intensity ratio Θ(1/d + n/(d·h) + 1/b) — the n/d term is cut by h (MQA, GQA). MQA-XXL ran 5.4× faster than MHA-XXL at equal quality; 8 groups is the GQA sweet spot.

Disaggregated serving: balance prefill:generate servers via P/prefill_lat = B·G/(gen_lat·median_decode) → ~3:1 for 8k prefill / 512 decode. Tokens evicted/step = B·(P+G)/G.

Speculative decoding (Leviathan et al.): expected tokens/step (1−α^(γ+1))/(1−α); walltime speedup (1−α^(γ+1))/((1−α)(γc+1)); wins whenever α > c. Cuts HBM reads (memory-bound win), raises FLOPs/concurrency. Measured 2–3×; empirical acceptance ~3–4 tokens (NL), more for code.

A.9 Batch-size & critical-batch heuristics

Threshold Value Meaning
Matmul compute-bound (per-replica tokens) 240 (v5e), 281 (B200), 295 (H100) below → memory-bound
Weight-only int8/fp8 120 halved
DP/FSDP min batch/device C/W ≈ 2550 (v5p 1-axis), ~850 (3-axis), ~2500 (H100), **~5625 (B200)** below → comms-bound
FSDP+TP min batch/chip 2550²/(2F) ≈ 100 the combined floor
TP max degree Y < F·W/C → ~8-way beyond → NVLink/ICI-bound
Cross-pod min batch/pod C/W_dcn (73,440 on v5p) below → DCN-bound
MoE generation batch 120·E/k (3840 for E=256,k=8) to keep experts compute-bound
Generation throughput use the largest per-chip batch up to B_crit KV memory permitting

Critical batch size (statistical, not roofline): ramp it. DeepSeek-V3 ramped global batch 3072 → 15360 over the first 469B tokens, peak LR 2.2e-4 (2k-step warmup) → 2.2e-5 cosine (DeepSeek-V3). LR–width coupling (Vaswani): lrate = d_model^(−0.5)·min(step^(−0.5), step·warmup^(−1.5)), warmup 4000.

A.10 Worked back-of-envelope templates (instantiate for your run)

Training time: T = 6·P·tokens / (n_chips · C · MFU). LLaMA-3 70B: 6·70e9·15e12 = 6.3e24 FLOPs; 8960-chip v5p pod @ 40% MFU → 44 days. On 4096 H100 @ 45% → ~40 days.

Min chips to fit (training): (2P + 8P + checkpoints)/HBM. LLaMA-3 70B 4M-batch → 140 GB params + 560 GB optimizer + 20.9 TB checkpoints = 21.6 TB → ≥225 v5p (but use large clusters for FLOPs, not memory). With Muon, optimizer term drops 560 → ~280 GB (one fewer fp32 buffer).

Min chips to serve: ceil(P·bytes_per_param / HBM) then round to a slice; add KV = (KV/token)·S·B. LLaMA-3 70B int8: 70 GB params + 41.9 GB KV (B=32, 8k) = 112 GB → 7 v5e → 4×2; step ≈ 112e9/(8·8.2e11) = 17 ms → 1882 tok/s.

For a 10T-total / 200B-active MoE on 100k GB200: sparsity E/k sets activated-param FLOPs = 6·200e9·tokens; total params 10T sets memory = 10e12·(2 + optimizer)/sharding. With Muon (~6 B/param) the parameter+optimizer footprint is ~60 TB, spread by EP×FSDP across the 100k-GPU / ~1389-node NVL72 fabric. Keep EP within an NVL72 domain (72 GPUs) where AllToAll is 900 GB/s, since cross-node AllToAll degrades >4×; stay compute-bound by holding per-GPU batch above C/W ≈ 5625 for DP/FSDP and 2250e12/900e9 = 2500 for TP, capping TP at ~8-way and pushing the rest into PP+EP. MoE generation batch must exceed 120·E/k to keep experts compute-bound.
