# Chapter 17 — Kernels for Blackwell: FlashAttention-4, Grouped GEMM, and the Fusion Inventory

Every rooflines-and-MFU argument downstream (Ch18, Ch23) presumes a kernel layer that actually delivers the FLOPs a spec sheet promises. That layer turned over completely between Hopper and Blackwell — not incrementally, but architecturally: Blackwell moved matrix-multiply issuance from warp-scope to single-thread CTA-scope, introduced a new on-chip memory space that must be managed by hand, and shifted the attention kernel's binding constraint from tensor-core FLOPs to shared-memory bandwidth and exponential-unit throughput. As of July 2026 the software response to that shift is real, fast-moving, and unevenly finished: FlashAttention-4 exists and is fast, but its FP8 path is forward-only; DeepGEMM already speaks Blackwell's native microscaling format while attention kernels do not; and the single hardest unsolved problem in the entire stack — grouped GEMM at the small-M shapes a 10T-total/200B-active MoE actually produces — has a documented 10x gap between naive and hand-tuned code, with the best-known public technique still absent from every major library's default path. This chapter inventories that layer precisely enough to plan a build-or-fund list against it.

## The Blackwell Programming Model: tcgen05, TMEM, and 2-CTA MMA

The single biggest conceptual change for a kernel author moving from Hopper to Blackwell is where matrix-multiply issuance lives. Hopper's WGMMA instruction is warpgroup-scoped — CuTe's own layout algebra expresses this as `ThrID = Layout<_128>`, 128 cooperating threads jointly issuing one WGMMA — so a kernel author reasons about pieces of the MMA result living in each thread's registers. Blackwell's `tcgen05.mma` collapses that to `ThrID = Layout<_1>`: a single thread issues the instruction on behalf of the entire CTA (per SemiAnalysis's ["Dissecting NVIDIA Blackwell Tensor Cores"](https://newsletter.semianalysis.com/), 2025–2026, corroborated by [NVIDIA's CUTLASS tcgen05 programming docs](https://docs.nvidia.com/cutlass/)). The practical consequence is that threads no longer implicitly own accumulator fragments in registers; the accumulator (and optionally operand A) now lives in **Tensor Memory (TMEM)**, a new 256KB-per-SM on-chip space wired directly to the tensor cores that software must explicitly allocate and manage — there is no implicit register-file fallback the way there was on Hopper and earlier.

The second load-bearing primitive is **2-CTA (2-SM) cooperation**, CUTLASS's "cluster launch" mechanism: two adjacent CTAs in the same cluster jointly execute one `tcgen05.mma` via `cta_group::2` in PTX (`2CTA` in SASS), with only the leader CTA issuing the instruction while both CTAs' TMEM and SMEM remain addressable within the pair. Microbenchmarks show near-perfect weak scaling — doubling compute resources roughly doubles throughput — and for SMEM-bound shapes with N<128, splitting operand-B SMEM bandwidth demand across two SMs delivers **over 2x speedup**, not just 2x compute. A companion primitive, **TMA multicast**, lets a single Tensor Memory Accelerator load push data into multiple CTAs' shared memory simultaneously via a hardware CTA bitmask, coalesced by the L2 Request Coalescer to approach the ideal `1/cluster_size` L2-bytes-per-SMEM-byte ratio. And because clusters are guaranteed co-scheduled within a GPC, Blackwell adds **Cluster Launch Control (CLC)**: dynamic, persistent-kernel work-stealing across SMs, so an SM that finishes its assigned tile early can grab another SM's remaining work rather than idling.

CLC is worth flagging as more than a footnote. The identical "persistent kernel plus atomic-counter-driven work-stealing" pattern recurs independently in three unrelated places in this chapter — FA4's tile scheduler, the state-of-the-art NVFP4 grouped-GEMM worklog below, and Blackwell's own hardware-level CLC feature. That convergence is evidence that irregular-workload load balancing (varlen attention, small-M MoE) is becoming a first-class, hardware-assisted primitive on this generation of silicon, rather than a software workaround layered on top of a fixed hardware model the way Hopper-era kernels solved the same problem with clever static tile-size heuristics. (The synthesis across sources is ours; no single source makes the three-way connection.)

One further architectural wrinkle matters for anyone assuming a single Blackwell code path: **consumer/workstation Blackwell (SM120/SM12x)** has a tensor-core programming model closer to Ampere's `mma.sync` than to datacenter SM100's `tcgen05`, forcing Triton, CUTLASS, FA4, and FlashMLA to each maintain three genuinely different code paths — Ampere-style, Hopper WGMMA, Blackwell tcgen05 — rather than one path with minor per-generation tweaks. A 10T/200B design living entirely on GB200/SM100 datacenter parts is insulated from this, but it explains why consumer-Blackwell kernel maturity (flagged below) lags so far behind datacenter Blackwell despite sharing an architecture name.

## FlashAttention-4: Redesigning Around Asymmetric Scaling

FlashAttention-4 (Zadouri, Hoehnerbach, Shah, Liu, Thakkar, Dao, ["Algorithm and Kernel Pipelining Co-Design for Asymmetric Hardware Scaling"](https://arxiv.org/abs/2603.05451), Mar 2026) is not FA3 ported to Blackwell; it is a redesign forced by one hardware fact. Going from H100 to B200, BF16 tensor-core throughput scales 2.25x (1 to 2.25 PFLOPS dense on the HGX B200 SKU), while shared-memory bandwidth and the count of Special Function Units executing softmax's `exp` stay essentially flat. The paper's own per-SM roofline (tile M=N=D=128) makes the shift concrete: in the forward pass, tensor cores need roughly 1,024 cycles and the exponential unit is *also* roughly 1,024 cycles — a genuine co-bottleneck — against 768 cycles of SMEM traffic. In the backward pass, which needs five MMAs instead of two, SMEM traffic balloons to roughly 3,328 cycles against 2,560 cycles of MMA compute, making shared-memory bandwidth the dominant backward-pass bottleneck by 25–60% depending on configuration (roofline independently corroborated by [Colfax Research's FA4 technical summary](https://research.colfax-intl.com/), 2026).

FA4's forward pipeline uses distinctly more warp roles than FA3's ping-pong scheme: a load warp driving TMA transfers, an MMA warp issuing async `tcgen05` matmuls, **two 4-warp softmax warpgroups** deliberately synchronized so they never compute exponentials simultaneously (spreading the SFU-plus-emulated-FMA exp load across time), a dedicated correction warpgroup applying rescaling off the critical path, and one to two epilogue warps — with one tile's MMA overlapping the adjacent tile's softmax (first documented in [Modal's reverse-engineering writeup](https://modal.com/blog/), Sept 2025, and confirmed by the paper).

Two specific micro-optimizations are worth naming because they are genuinely novel, not incremental. First, FA4 decomposes `2^x` via Cody-Waite range reduction and approximates the fractional part with a **degree-3 polynomial in Horner form** (coefficients `1.0 + 0.6951x + 0.2276x² + 0.0771x³`, fit via Sollya for minimal relative error), running this software path on the underused FMA datapath *in parallel with* the hardware `MUFU.EX2` instruction — splitting the softmax exponential workload across both rather than letting one displace the other. Second, **conditional softmax rescaling**: standard FlashAttention rescales the running output accumulator every time a new K-block potentially raises the row max; FA4 defers correction until the new max exceeds the old one by more than a tunable threshold τ (the paper cites τ ≈ log₂(256) = 8.0 in one configuration), reported to cut rescale operations by roughly **10x** (the mechanism was independently corroborated by [SemiAnalysis's live coverage](https://x.com/SemiAnalysis_) of Tri Dao's Hot Chips 2025 talk).

The performance headline is consistent across primary and secondary sources: **1,605–1,613 TFLOPS/s BF16**, 71% of B200's ~2,250 TFLOPS BF16 dense peak (the HGX B200 SKU — see the SKU caveat below), 1.1–1.3x faster than cuDNN 9.13, and 2.1–2.7x faster than Triton, with the largest wins at 4K+ sequence lengths. One conflation worth flagging for accuracy: a commonly repeated "~1.2 PFLOPS/s on B200" figure is actually FA3's FP8 number on H100 ([arXiv:2407.08608](https://arxiv.org/abs/2407.08608), Jul 2024), not an FA4/B200 figure — FA4's own reported BF16 peak is ~1.6 PFLOPS, not 1.2.

The "71% of B200's peak" figure itself carries the SKU trap flagged repeatedly elsewhere (Ch10, Ch16, Ch18): the FA4 paper's denominator is the **HGX B200** dense BF16 peak of 2,250 TFLOPS, not GB200 NVL72's dense BF16 peak of 2,500 TFLOPS established in Ch16's reconciliation. Against the GB200 NVL72 denominator, FA4's measured 1,605–1,613 TFLOPS/s reads as **64.2–64.5%** of peak rather than 71% — the kernel's absolute throughput is unchanged, but anyone porting "71%" into a GB200-denominated roofline calculation should use the corrected figure. To be precise about provenance: the 64.2–64.5% is our own division (1,605–1,613 ÷ 2,500), applied by analogy with Ch16's Megatron-Core MFU correction — the FA4 paper itself never flags its "71%" as SKU-ambiguous, and no source states the corrected percentage.

## FA4's Backward Pass and the July 2026 Repo-Verified Feature Matrix

Because papers lag shipped code, the state of FA4 as actually usable in a training stack is best established by reading the repository directly rather than the paper. A direct inspection of [`Dao-AILab/flash-attention`](https://github.com/Dao-AILab/flash-attention) (`flash_attn/cute/`, HEAD `0816ef1`, pulled July 13, 2026) against the March 2026 paper shows real, dated progress.

**Backward pass**: fully present as of the paper, a 5-MMA chain (recompute S, compute dP, dV, dS, then dQ/dK) bottlenecked by SMEM bandwidth rather than compute, with 2-CTA mode as the main backward-specific optimization and a deterministic mode reaching **85–90% of nondeterministic throughput** via semaphore-locked, shortest-processing-time-first accumulation scheduling. This is a genuinely fast maturation: Modal's engineers, reverse-engineering the kernel in September 2025, wrote that "a backwards version of this kernel is not available, but is expected imminently" — meaning FA4 went from no backward pass at all to a full, deterministic-capable backward in roughly five to six months.

The July 2026 repository confirms varlen (`cu_seqlens_q/k`, `seqused_q/k`, a `SeqlenInfo` dataclass precomputing tile offsets once per batch element — this is the THD/packed-sequence path Ch15's dataloading architecture depends on), full GQA via `pack_gqa.py`, causal/local/sliding-window and arbitrary score/mask modifiers unified through a `_resolve_causal_local_window()` helper, first-class block-sparsity (`block_sparsity.py`), and dedicated sparse-MLA forward and backward kernels for DeepSeek V4's attention variant. The sparse-MLA backward landed via release `fa4-v4.0.0.beta19`, published June 24, 2026 — roughly two months after DeepSeek-V4's April 2026 release, and about three weeks before the July 13 evidence snapshot. Head-dim-256 gets its own hand-tuned 2-CTA forward/backward path rather than falling through the generic kernel (release dates via the [GitHub Releases feed](https://github.com/Dao-AILab/flash-attention/releases)).

Two gaps remain load-bearing for a training-time precision decision. **FP8 is explicitly forward-only**: the code raises `NotImplementedError("FA4 CuTe FP8 backward is not supported yet (forward-only).")` whenever `is_fp8 and requires_grad`, and further asserts FP8 is SM100-only — there is no FP8 attention *training* path on Blackwell in FA4 CuTe as of July 2026. And a repository-wide grep for "mxfp8", "nvfp4", or "block_scale" across `flash_attn/cute/*.py` returns zero hits: FA4's FP8 support is plain per-head-batch descale scalars (`q_descale`/`k_descale`/`v_descale`), not Blackwell's native 32-element microscaling format.

Why the backward specifically is the holdout deserves a mechanism, not just a status report, because the mechanism predicts how long the gap will last. (The argument here is our own; the FA3 facts it builds on are cited.) FP8 attention was already hard in the *forward* direction — FA3's FP8 forward needed block quantization plus in-kernel "incoherent processing" (a Hadamard-transform rotation of Q and K to spread outlier mass across dimensions) just to keep softmax inputs inside E4M3's dynamic range ([arXiv:2407.08608](https://arxiv.org/abs/2407.08608)). The backward is strictly worse on every axis that made that necessary. Its central intermediate, dS = P ∘ (dP − D), is a product of a probability (spanning many orders of magnitude across a row by construction — that is what softmax *does*) with a centered gradient term, so its dynamic range is wider than anything in the forward pass; and where the forward's quantization error stops at the output, the backward chains five MMAs, so a quantization error in dS propagates multiplicatively into dQ, dK, *and* dV, and from there into every step of training rather than one activation. An FP8 attention backward is therefore a numerics-design problem (which tensors can survive 8 bits, with what scaling granularity, and where the chain must promote) before it is a kernel-engineering problem — which is why it has stayed unshipped while far more elaborate *forward* kernels (sparse MLA, block-sparse masks) landed in the same repository, and why a training team should not model the gap as "a few engineer-months behind the forward path."

This is a real asymmetry within the same 2026 Blackwell kernel stack: DeepGEMM (below) already speaks native UE8M0 block-scaling for MoE GEMM, while FA4 attention does not — the MoE-GEMM and attention halves of a Blackwell-native precision stack are on different format-maturity curves, a fact that only becomes visible by diffing the two codebases directly rather than reading either project's documentation in isolation. Release cadence is roughly weekly (beta8 through beta21, April 8 through July 8, 2026 — thirteen releases in thirteen weeks), meaning any "current status" claim here is dated precisely and likely to be stale within weeks. The repository also depends on `quack.compile_utils`, meaning FA4 and Quack (Dao-AILab's separate CuTe-DSL library for memory-bound ops — §The Fusion Inventory below) share build infrastructure and their release cadences correlate.

## cuDNN SDPA: Closing the Gap From Behind

cuDNN was first to ship *any* performant Blackwell attention kernel, ahead of both FA3-for-Blackwell (which never existed — WGMMA does not exist on SM100) and FA4, during a 2025 window practitioners describe as having "a lot of flux in attention implementations for new Blackwell hardware." By FA4's own account, NVIDIA and the FA4 authors collaborated starting with cuDNN 9.13/9.14 to fold FA4's techniques back into cuDNN, and "the latest cuDNN 9.19.1.2 now offers similar performance to FA4." cuDNN's own 9.19-line [release notes](https://docs.nvidia.com/deeplearning/cudnn/) document a unified SDPA-forward backend descriptor, GPT-OSS-style attention-sink support for forward and backward on all SM80+ GPUs, and — the detail that should give a frontier-run planner pause — a fix for **MXFP8 SDPA-backward NaN/mismatch bugs specifically on Blackwell**.

The net kernel-selection calculus for 2026: cuDNN and FA4 are close enough in raw BF16 dense/causal throughput (FA4 still leads by roughly 10–30% forward) that the practical decision is more about integration convenience — cuDNN ships in every NGC container, FA4 CuTe is a separate, fast-moving pip package on a weekly release cycle — than about a large performance gap. But the recency of the MXFP8 backward correctness fix implies that code path is young and under-exercised at production scale; treating "cuDNN supports MXFP8 attention" as a settled checkbox for a $2B run's precision recipe would be premature.

## Triton on Blackwell: Why Gluon Exists

Triton can *emit* `tcgen05` instructions — generic `tcgen05.cp` lowering and matrix descriptors landed by early 2026 — but multiple independent sources converge on the same conclusion: matching hand-tuned or CuTe-DSL-quality Blackwell attention codegen from Triton's high-level, automatically-pipelined abstractions failed. Blackwell's single-thread MMA issuance (with its leader-election requirement), multi-barrier pipelining, and TMEM allocation-with-contention-handling are exactly the low-level, hard-to-discover patterns a block-level compiler struggles to infer automatically (see [Patrick Toulme's "CuTile on Blackwell: NVIDIA's Compiler Moat Is Already Built"](https://patricktoulme.substack.com/), 2026).

The Triton team's response was **Gluon**: a lower-level, Triton-adjacent frontend where the programmer manually allocates SMEM buffers, places barriers, and issues async-copy and `tcgen05`-MMA instructions explicitly — trading Triton's automatic pipelining for CuTe-DSL-like explicit control while staying inside the Triton/MLIR toolchain. Gluon tutorials, including a dedicated `06-tcgen05.py` walkthrough and a documented 2-CTA blocked-scaled-matmul example, live directly in the [Triton repository](https://github.com/triton-lang/triton), and a third-party worked example (["Inside a Blackwell 2-CTA GEMM: A Gluon Kernel Tour"](https://tanmaypatil.com/), 2026) demonstrates the pattern is reproducible outside NVIDIA and Dao-AILab, not an internal-only workaround. The severity here is worth stating plainly: this was not an incremental compiler improvement but an entirely new, lower-level frontend spun up specifically because high-level Triton abstractions could not close the gap — a stronger statement about tile-based-compiler limits on Blackwell than "Triton is a bit behind cuDNN."

GPU MODE's own lecture catalog corroborates this framing from the authors' side: Lecture 104, "Gluon: Tile-Based GPU Programming with Low-Level Control" (Peter Bell, Mario Lezcano, Keren Zhou), is the primary talk from Gluon's own authors ([github.com/gpu-mode/lectures](https://github.com/gpu-mode/lectures)). A direct correction to a common assumption belongs here: GPU MODE's roughly 106-lecture catalog as of mid-2026 has almost no Blackwell/DeepGEMM-specific frontier-lab kernel disclosures — Tri Dao's FA4 talk was delivered at Hot Chips 2025, not GPU MODE, and no dedicated DeepGEMM lecture exists by title search. GPU MODE is the stronger source for kernel-authoring *pedagogy* — CuTe, Triton, Gluon fundamentals — while frontier-lab kernel disclosures themselves cluster at Hot Chips and in papers and repos directly. The pedagogically load-bearing lectures for anyone building this stack are Lecture 57 ("CuTe," Cris Cecka, the conceptual on-ramp), Lecture 86 ("Introduction to CuTeDSL," Vicki Wang, explicitly framed as onboarding for an NVIDIA NVFP4/Blackwell kernel-writing competition), Lecture 103 ("Fundamentals of CuTe Layout Algebra," Jack Carlisle and Jay Shah — Shah is a Colfax co-author of the FA3 and CUTLASS-Blackwell tutorial material cited above), Lecture 84 ("Numerics and AI," Paulius Micikevicius, NVIDIA's FP8/Transformer Engine architect), and Lecture 23 ("Tensor Cores," Vijay Thakkar and Pradeep Ramani, both NVIDIA CUTLASS-team).

## CuTe-DSL Adoption, ThunderKittens, and the Tile-Language Field

**CUTLASS 4.x's CuTe-DSL** — a Python authoring surface for the primitives above — is now the primary kernel-authoring layer for Blackwell at NVIDIA and Dao-AILab: FlashAttention-4 itself is written in CuTe-DSL rather than CUTLASS C++, DeepGEMM explicitly builds on CUTLASS/CuTe concepts, and Dao-AILab's **Quack** library ([github.com/Dao-AILab/quack](https://github.com/Dao-AILab/quack), Jul 2025) reimplements RMSNorm, softmax, LayerNorm, and cross-entropy purely in CuTe-DSL, claiming 33–50% faster than `torch.compile` plus Liger on H100 at roughly 3TB/s memory throughput — near speed-of-light for memory-bound kernels. Adoption goes further than expert tooling: PyTorch and NVIDIA shipped a **fourth TorchInductor autotuning GEMM backend built on CuTe-DSL** in 2026, alongside Triton, CUTLASS C++, and cuBLAS, beating Triton by up to **1.73x for BF16, 1.78x for MXFP8, and 1.6x for NVFP4** on B200 in decode-shaped (small-M) GEMMs, and covering `grouped_mm` directly ([pytorch.org/blog/gemms-torchinductor-cutedsl-backend](https://pytorch.org/blog/)). The underappreciated implication: a plain `torch.compile(mode="max-autotune")` call on current PyTorch can already silently select a CuTe-DSL-compiled kernel over both Triton and hand-written CUTLASS C++ for an ordinary matmul on B200, without the user ever touching CuTe-DSL syntax. Status, per NVIDIA's own docs: **public beta, targeted to graduate by end of summer 2026** — any "production-ready" framing is premature by NVIDIA's own stated timeline as of mid-2026.

Stanford's **ThunderKittens**, an embedded-C++ tile-primitive DSL, shipped Blackwell support in March 2025 and a major 2.0 rewrite in February 2026 adding MXFP8/NVFP4 support, Cluster Launch Control scheduling, and explicit TMEM control; GEMM kernels are reported to "surpass or match cuBLAS" on B200 across BF16/MXFP8/NVFP4, with attention forward and backward running near cuDNN speed, "up to 2x faster than FA3 on H100" — note this last figure compares against FA3-on-H100, not FA4-on-B200 ([Hazy Research blog](https://hazyresearch.stanford.edu/blog/), Feb 2026). The 2.0 release's engineering post-mortems are unusually concrete and worth citing for their specificity: removing unnecessary memory-fence instructions recovered **~20 TFLOP/s** in GEMM kernels; discovering that `tcgen05.cp` pipelines implicitly with `tcgen05.mma` with no explicit sync needed recovered **~500 TFLOP/s (~10%)** for NVFP4; and replacing manual leader-thread-selection loops with PTX's `elect.sync` instruction removed serialization and improved small-GEMM throughput by up to 10%. These are exactly the kind of micro-optimizations invisible in a paper abstract that determine whether a hand-written kernel hits 80% or 95% of a competitor's throughput.

Two further tile-language efforts round out the field and are easy to conflate. **cuTile**, NVIDIA's own tile-based Python DSL shipped in CUDA 13.1/13.2, has its `tileiras` compiler backend supporting only Blackwell and Ampere/Ada as of mid-2026 — Hopper support is explicitly deferred to "coming versions" ([NVIDIA's cuTile blog](https://developer.nvidia.com/blog/)). **Tilus** ([github.com/NVIDIA/tilus](https://github.com/NVIDIA/tilus)), a separate Triton-inspired language emphasizing explicit SMEM and register control closer to Gluon's philosophy than Triton's, spans both Hopper and Blackwell today with step-by-step tutorials reaching cuBLAS-level performance. The practical consequence for a shop that must span an H100/H200 fleet alongside a GB200 fleet simultaneously: cuTile currently cannot, Tilus and CuTe-DSL both can. On the JAX side, Pallas's Mosaic-GPU backend documents Blackwell matmul support — warp specialization, tiled epilogues, collective 2-CTA matmul, persistent kernels with CLC ([JAX Pallas Blackwell matmul docs](https://docs.jax.dev/en/latest/pallas/gpu/blackwell_matmul.html)) — meaning JAX is not architecturally blocked from the same primitives CUTLASS/CuTe-DSL expose. No independent benchmark comparing Mosaic-GPU-authored Blackwell kernels against CUTLASS/CuTe-DSL or Triton/Gluon equivalents appears to exist publicly; treat Mosaic-GPU's relative maturity as a genuine open question for a JAX-based stack (see Ch19).

## Grouped GEMM at MoE Expert Shapes: The Small-M Problem

This is the single most consequential kernel gap for a 200B-active MoE design, and it is worth stating the mechanism precisely. A grouped GEMM for MoE shares K (hidden size) and N (expert intermediate size) across experts but gives each expert a different, data-dependent M — the number of tokens actually routed to it. At the expert-parallelism degree a 10T-total model needs, per-GPU per-expert M can be small, and small-M GEMMs are architecturally penalized on Blackwell specifically: TMEM occupancy caps at **one CTA per SM** once the accumulator plus scale-factor tensors fill the 512-column TMEM partition, so a naive per-expert-kernel-launch approach burns enormous time in launch and synchronization overhead relative to actual compute.

A detailed public worklog on NVFP4 grouped-GEMM optimization on B200 ([mufeezamjad.com](https://mufeezamjad.com/), 2026) is unusually concrete about the resulting headroom. A reference/naive implementation runs at **238µs geomean** across four representative expert shapes; a hand-optimized version using a persistent kernel with a single flattened global work queue (atomic-counter-based tile scheduling across all experts, replacing per-expert launches), 4-stage warp-specialized pipelining (up from 2), TMA cluster multicast for operand-A reuse (up to 4x traffic reduction on memory-bound shapes), and problem-aware L2 cache-eviction hints reaches **23.8µs geomean — roughly 10x** — with individual shapes improving 7.4x to 13.9x. Even this optimized kernel only achieves **~50% average / ~80% steady-state tensor-core utilization** and **12.4% SM occupancy** (the TMEM-fits-one-CTA constraint again), and the author explicitly notes the best-known public submissions (~16µs geomean) go further via cooperative `cta_group::2` MMA — 2-CTA-shared TMEM, amortizing per-CTA overhead — and an M-major epilogue output layout enabling 256-bit vectorized stores, techniques the author's own kernel does not implement. The competition-versus-production-library provenance of that ~16µs figure is itself unresolved — the source's language ("faster submissions") implies a competitive kernel-optimization context rather than a shipped default, meaning a training team may need to reimplement the technique rather than `pip install` it.

How small is small-M for *this* design, in training rather than decode? Worth pinning down, because the answer is worse than the tail-risk framing suggests. (The derivation is ours; Ch18 carries the full layout math.) The average number of tokens an expert's GEMM sees per microbatch is (global batch) × k/E ÷ (DP degree × microbatch count) — and at the bulk-training layout Ch18 recommends (global batch 10–15M tokens, DP=192, m≥18 microbatches for pipeline-bubble control, 512 experts top-8), that works out to roughly **M ≈ 45–70 tokens per expert per grouped-GEMM call, as the average case, before any routing imbalance**. Compare the one directly measured datapoint at a nearby shape (Ch10): M=128 (N=4096, K=7168) delivered just 190 TFLOPS on H800 — about 12% of the same library's well-shaped peak. DeepSeek-V3 never lived in this regime, not because its experts are wider but because its DP degree was 2: its per-replica batches were enormous, so per-expert M stayed in the thousands. Scaling to 100k GPUs at a fixed critical-batch-bounded global batch *forces* DP into the hundreds and starves every expert's M proportionally — the small-M problem is the average operating point of this design, not an unlucky corner, and Ch18 works the consequences (and the escape routes: bigger global batch, fewer/larger microbatches, higher EP) into the layout math directly.

The takeaway for a 10T/200B design is unambiguous: off-the-shelf grouped GEMM at production MoE expert-parallelism degrees on Blackwell is closer to "a documented 10x of headroom exists between naive and hand-tuned" than "solved," and the state-of-the-art technique — 2-CTA cooperative MMA specifically for grouped GEMM — is not in any public library's default path as of mid-2026.

**DeepGEMM** ([github.com/deepseek-ai/DeepGEMM](https://github.com/deepseek-ai/DeepGEMM), inspected directly, July 13, 2026) supports SM90 and SM100; SM90 grouped and dense GEMM is NT-layout only, while SM100 supports all four layouts (NT/TN/NN/TT) and requires LHS scaling factors in packed UE8M0 format — DeepGEMM already speaks Blackwell's native block-scaling format for FP8, the asymmetry with FA4 attention noted above. Two grouped-GEMM variants ship: a contiguous layout (tokens sorted by expert, M-axis grouped) for training-style large-batch MoE, and a masked layout for inference-decode-style variable-length CUDA-graph use. Cited peak is **1,550 TFLOPS on H800** (Apr 2025); DeepSeek itself has published no Blackwell-native peak figure for DeepGEMM — a third-party guide's claim of "another 1.5–2x on top of H200" is not DeepGEMM's own number and should be treated as an estimate. The 2026-era changelog adds "Mega MoE" (overlapped NVLink communication fused with grouped GEMM — the mechanism behind DeepSeek-V4's wave-based EP overlap discussed in Ch4), an FP4 indexer kernel, and FP8×FP4 mixed-precision GEMM.

**CUTLASS** ships dedicated grouped-GEMM examples (`75_blackwell_grouped_gemm`, `75_blackwell_grouped_gemm_block_scaled.cu`) claiming up to **5x throughput over H200-FP16** using Blackwell FP4. But this maturity is uneven across Blackwell SKUs in a way worth flagging for anyone tempted to treat "CUTLASS supports it" as a checkbox: consumer/workstation Blackwell (SM120/SM121, RTX Pro 6000, DGX Spark) has documented correctness bugs — [flashinfer-ai issue #2723](https://github.com/flashinfer-ai/flashinfer) reports CUTLASS Grouped Block-Scaled GEMM producing invalid output on SM120 NVFP4 MoE, and a vLLM issue (#3628) confirms CUTLASS 4.5 lacks a `CollectiveBuilder` specialization for tensor/token-scaled FP8 grouped GEMM on SM_120/SM_121 entirely. This is a distraction for a datacenter-only GB200 design (SM100 throughout), but it is evidence CUTLASS's grouped-GEMM code is newer and less battle-tested than its dense-GEMM code even in places it nominally "supports" a target.

**Transformer Engine**, per Megatron-Core's own 2026-Q2 roadmap ([Megatron-LM issue #4815](https://github.com/NVIDIA/Megatron-LM/issues/4815)), lists "GroupedGEMM and Gradient Accumulation Fusion," "GroupGEMM + SwiGLU + quantize fused MLP," and "grouped GEMM BF16/MXFP8 support with CUDA Graph compatibility" as completed items for the June 2026 (26.06) Megatron-Core release, alongside FP8/MXFP8 grouped-GEMM support and FP8 weights with BF16 optimizer states — confirming TE's grouped-GEMM plus fused-MLP path is production-integrated into the reference training framework as a pluggable backend alongside DeepGEMM.

| Library | SM100 layouts | Native Blackwell block-scaling | Best documented small-M result | Known gap |
|---|---|---|---|---|
| DeepGEMM | NT/TN/NN/TT | Yes (UE8M0, packed) | 1,550 TFLOPS/H800 at well-shaped M; no published B200 peak | No confirmed Blackwell-native peak figure |
| CUTLASS (`75_blackwell_grouped_gemm`) | via examples | Yes (block-scaled example) | up to 5x over H200-FP16 (FP4, well-shaped) | SM120 correctness bugs; grouped-GEMM path newer/less battle-tested than dense |

Megatron-Core's own MoE decomposition — route, permute, dispatch (all-to-all across EP ranks), grouped GEMM, combine, unpermute — surfaces a related, sobering number about where time actually goes in MoE training. The DeepEP token dispatcher fuses permute into dispatch and unpermute into combine as single fused kernels, and a new HybridEP dispatcher targets GB200 specifically ([Megatron-Core MoE user guide](https://docs.nvidia.com/megatron-core/)). But even after this optimization, the Megatron-Core MoE technical report ([arXiv:2603.07685](https://arxiv.org/abs/2603.07685), Mar 2026) states that **routing and permutation overhead adds ~9% to layer execution time**, and — the more consequential figure — **GEMMs occupy under 50% of execution time for DeepSeek-V3-scale MoE training, versus roughly 70% for a dense Llama-3-405B run**: for current-generation MoE stacks, roughly half of step time is router, permute, communication, and other non-FLOP-bearing kernel overhead, not the matmuls themselves. The same report gives 1,233 and 1,048 TFLOPS/GPU sustained on GB300 and GB200 respectively for DeepSeek-V3-685B, and 974 and 919 TFLOPS/GPU on GB300 and GB200 respectively for Qwen3-235B. The once-open question of which precision those figures are measured in is now resolved: the report's own Table 11 Dtype column and appendix mark these runs as **MXFP8**, so the correct denominator is GB200's dense FP8/MXFP8 peak of 5,000 TFLOPS — giving **21.0% MFU on GB200 and 24.7% on GB300 for DeepSeek-V3-scale MoE training** (and 18.4%/19.5% for Qwen3-235B), not the 41.9% "BF16 MFU" that dividing by the 2,500 BF16 peak would imply. The corrected figure is the closest thing to a citable, real-world, kernel-stack-level sustained-MFU number for GB200 MoE training available anywhere public, and it is reassuringly consistent with DeepSeek-V3's own H800 run at 19.5% FP8-normalized MFU — MoE training in the low twenties of percent, across two hardware generations and two independent stacks, is the real baseline the maturation curve starts from (Ch23 tracks where the 26.06-era containers have since pushed it).

## FlashMLA, DSA, and the Varlen/THD Chain That Gates Packing

**FlashMLA** ([github.com/deepseek-ai/FlashMLA](https://github.com/deepseek-ai/FlashMLA)) covers SM90 (H800/H100) dense BF16 decode, sparse FP8-KV decode, and sparse prefill, with SM100 (B200) adding dense MHA prefill on top of the SM90 feature set. Measured numbers: SM90 dense MLA decode up to 3,000 GB/s (memory-bound) or 660 TFLOPS (compute-bound); SM90 sparse MLA decode (FP8 KV) 410 TFLOPS; **B200 sparse MLA decode ~350 TFLOPS**, which the repository's own README candidly flags as "not really optimized yet"; B200 sparse MLA prefill up to 1,450 TFLOPS; B200 dense MHA prefill forward up to 1,460 TFLOPS, backward 1,000 TFLOPS. KV-cache quantization uses FP8 E4M3 with per-128-token scales, keeping the RoPE component unquantized in BF16. A self-reported gap from the maintainers themselves is a rarer and more trustworthy kind of evidence than most vendor claims in this chapter.

Native Sparse Attention (NSA, the block-wise research design) and DeepSeek Sparse Attention (DSA, the shipped V3.2/V4 design using a finer-grained, token-wise "lightning indexer") are related but distinct, and DSA's kernels are genuinely open across two tiers: TileLang implementations (readable, research-oriented) released alongside DeepSeek-V3.2-Exp in September 2025 ([vLLM's coverage, 2025-09-29](https://vllm.ai/blog/)), and production CUDA kernels split across DeepGEMM (indexer logit computation) and FlashMLA (the sparse-attention kernels themselves). A third-party TileLang reimplementation of the DSA lightning-indexer training operator exists at [lemyx/tilelang-dsa](https://github.com/lemyx/tilelang-dsa) — evidence of community reproduction beyond DeepSeek's own release. DeepSeek-V4 (April 2026, full architectural detail in Ch4) extends the indexer to 64 indexer heads, 4x key compression, and per-query top-k of 512–1024 tokens at 1M-token context.

**Varlen/THD support is the mechanism that makes sequence packing (Ch15) viable at all**, and it is worth stating why this chapter and not the dataloading chapter owns the kernel-level detail: packing only pays off if the attention and MoE kernels consuming packed sequences correctly track cumulative sequence-length offsets rather than materializing padded tensors. Transformer Engine's THD format (`[total_tokens, num_heads, head_dim]`) relies on `cu_seqlens` propagated through TE's fused attention kernels; TE v2.16 specifically added `pad_between_seqs` support for both non-context-parallel and context-parallel (ring/A2A and P2P variants) combined with FA3 plus THD attention, alongside FA4 support for head_dim=256. A known load-imbalance wrinkle recurs here identically to the small-M grouped-GEMM problem above: with varlen, SM tile assignments naturally have different mainloop lengths — a long sequence's tile does more work than a short sequence's tile — which is exactly the load-balance problem persistent, atomic-counter-based scheduling is designed to fix. That the same architectural answer (LPT/persistent scheduling) recurs across FA4's tile scheduler, small-M MoE grouped GEMM, and varlen attention is evidence this is becoming the general-purpose 2026 Blackwell kernel-engineering pattern, not a one-off trick specific to any single kernel.

On the multi-token-prediction and hybrid-attention side, FlexAttention's FA4 backend natively supports sliding-window as a built-in mask type alongside causal, ALiBi, document-mask, and soft-cap, with measured GB200 speedups over Triton of 1.4–2.1x forward and 1.8–2.2x backward for sliding-window patterns specifically, and up to 2.7x forward / 3x backward for document masks. Separately, Megatron-Core's 2026-Q2 roadmap lists "GDN kernel fusion on Blackwell" (marked "working") and "pre-gated delta rule kernel fusion" as in-progress items tied to Qwen3.5-style hybrid linear-plus-full-attention architectures (Ch7) — hybrid-attention kernel fusion is an acknowledged, actively-being-built gap, not a finished item, in the reference MoE training framework as of mid-2026.

## The Fusion Inventory: Norm, RoPE, SwiGLU, Cross-Entropy, Optimizer

Below the attention and grouped-GEMM layer sits a set of smaller fusions whose aggregate effect on memory footprint and wall-clock is large enough to be load-bearing at frontier scale even though no individual fusion is architecturally novel.

**Norm/RoPE/SwiGLU fusion** is a solved, well-measured problem: aggregated fine-tuning-framework benchmarks report Triton-fused-kernel speedups of roughly **7x for RMSNorm, 5x for SwiGLU, and 2.3x for QK-RoPE fusion** relative to naive eager PyTorch (secondhand aggregates, not one citable benchmark), with the underlying pattern — normalizing `x + residual` without materializing the intermediate sum — standard across Liger-Kernel and Quack-style libraries. Fusing SwiGLU with FP8 amax/quantization specifically achieves latency "nearly identical" to standalone unquantized SwiGLU while producing FP8-quantized output directly — meaning the quantization step is effectively free when fused, which matters for any FP8-recipe pretraining stack (Ch10) where every activation needs a cast somewhere in the pipeline.

**Chunked cross-entropy is the single highest-leverage, most completely solved fusion in this inventory**, and it becomes non-optional rather than merely nice-to-have at the vocabulary sizes a 10T-total/200B-active multilingual/code design is likely to use. Apple's Cut Cross-Entropy ([arXiv:2411.09009](https://arxiv.org/abs/2411.09009), Nov 2024) computes only the correct-token logit plus an on-the-fly log-sum-exp over the vocabulary via a flash-memory-style kernel, never materializing the full [tokens × vocab] logit matrix; for Gemma-2 2B, this drops the loss computation's memory footprint from **24GB to 1MB** and total classifier-head training memory from **28GB to 1GB**, with no measured throughput or convergence regression. [Liger-Kernel](https://github.com/linkedin/Liger-Kernel)'s independent fused-linear-cross-entropy implementation reports roughly **5x memory reduction at vocab=163,840** via online-softmax chunked computation, and more broadly **40–55% peak-VRAM savings for vocab sizes above 100K**. The tradeoff is real, not free: the most memory-aggressive chunking configuration is reported to **more than double wall-clock latency** for the cross-entropy step specifically, since chunk count trades memory for latency directly. At frontier-scale vocabularies commonly exceeding 150–200K tokens, an unfused cross-entropy over a large batch-times-sequence-length would materialize a genuinely enormous logit tensor — this fusion is the difference between the loss computation fitting in memory at all, not a marginal optimization.

**Fused optimizer**: Apex's `FusedAdam` fuses the elementwise Adam update into one kernel and uses multi-tensor apply to batch the update across an entire model's parameter list into one or a handful of launches rather than one per tensor. No evidence of a CuTe-DSL or `tcgen05`-specific rewrite of the fused-optimizer kernel appears in any public source — this remains a straightforward memory-bound elementwise kernel where Hopper-era code likely runs unmodified on Blackwell, unlike the attention and grouped-GEMM kernels above, which needed ground-up redesign.

## torch.compile Coverage and the Fusion Ceiling

TorchTitan (Ch19) uses regional compilation — `torch.compile` applied per-`TransformerBlock` rather than to the whole model graph — specifically to keep FSDP2/TP-compatible graphs free of graph breaks and to exploit the fact that structurally identical transformer layers only need compiling once ([arXiv:2410.06511](https://arxiv.org/abs/2410.06511), 2024; still current practice per 2026 accounts). A compiled transformer block typically collapses from roughly 30 eager-mode kernel launches down to 5–8 fused kernels per layer: attention delegates to FA4 or cuDNN, matmuls delegate to cuBLAS or the CuTe-DSL Inductor backend with autotuned tile shapes, and layernorm collapses into a fused norm-residual op.

What still needs hand-authored kernels in 2026, and is unlikely to stop needing them: attention (Inductor correctly delegates to FA4/cuDNN rather than attempting its own codegen, given the specialization complexity documented above), grouped GEMM and MoE dispatch-combine (no evidence Inductor auto-fuses these — Megatron-Core and TE ship them as explicit hand-written ops), and cross-entropy/CCE-style vocabulary chunking (the online log-sum-exp restructuring is an algorithmic change, not a mechanical fusion a pattern-matching compiler would discover). This is a stable, three-item boundary between what a 2026-era compiler can find automatically and what an infra team must still own by hand.

## The Gap List: What a 10T/200B Run Must Build or Fund

Every item below is a documented gap, not a speculative one — each has a primary-source or repo-verified basis cited above.

- **FA4 FP8/MXFP8 backward for attention.** FA4 CuTe's FP8 path is forward-only (`NotImplementedError` in the current repo); a training team wanting FP8 attention with a working backward must either wait for this to land, fall back to BF16 attention with FP8 elsewhere in the model, or fund the backward kernel directly.
- **Blackwell-native block-scaled (MXFP8/NVFP4) attention on the training path.** No `mxfp8`/`nvfp4`/`block_scale` code exists anywhere in FA4 CuTe as of July 2026; cuDNN's MXFP8 SDPA-backward path has had recent Blackwell-specific correctness bugs. This is a bigger lift than FP8 backward alone — it needs a microscaling-aware softmax/rescaling redesign, not just a precision-format swap.
- **Production-grade cooperative 2-CTA MoE grouped GEMM in the public stack.** The 10x gap between naive and hand-tuned NVFP4 grouped GEMM at MoE expert shapes is documented; the technique closing the remaining gap to the best-known public results (2-CTA cooperative MMA, M-major epilogue) is in no major library's default path — this is the single highest-value kernel-engineering investment on this list given that GEMMs already occupy under 50% of MoE layer time even before accounting for this shape-efficiency loss.
- **A non-Gluon, high-level-Triton path to competitive Blackwell attention performance**, if the stack has a reason to avoid CuTe-DSL/Gluon's lower-level programming burden — as of July 2026 this does not exist; Gluon's explicit-control model is the only Triton-toolchain path that works.
- **CuTe-DSL graduating out of public beta**, with a rigorous C++-parity benchmark suite — NVIDIA states "beta to GA by end of summer 2026" with no committed calendar date published; a run committing to CuTe-DSL as production infrastructure before that graduation is committing to beta-quality tooling.
- **A Blackwell-native DeepGEMM peak-TFLOPS figure from DeepSeek itself.** The only DeepGEMM-published peak is 1,550 TFLOPS on H800 (Apr 2025); every B200 figure in circulation is a third-party estimate, not a DeepSeek-published number — worth closing before betting a cost model on it.
- **Hybrid/linear-attention kernel fusion (GDN, pre-gated delta rule) on Blackwell**, currently in-progress rather than finished in Megatron-Core's own roadmap — relevant to the extent the architectural strawman in Ch7 adopts a hybrid-attention design.
- **A microbenchmark of `tcgen05`'s FP8/MXFP8 accumulator width.** DeepSeek's promote-to-CUDA-core-FP32-every-128-elements workaround (Ch10) exists because Hopper's tensor cores accumulate FP8 partial sums at roughly 14 bits, not full FP32 — a measured fact with a measured ~2% GEMM-error consequence. Whether Blackwell's redesigned tensor-core path (TMEM-resident accumulators, hardware-applied microscaling factors) accumulates at full precision or inherits a truncated-mantissa fast path is not stated in any public source, and the answer decides whether the promotion workaround — and its interaction with the persistent-kernel scheduling patterns above — must be carried into every Blackwell FP8 GEMM or can be retired. This is a one-week microbenchmark (the Hopper measurement methodology transfers directly) that gates a precision-recipe decision worth far more; run it before porting any FP8 recipe.

## Implications for the 10T/200B Run

The kernel layer for this design is neither a solved commodity nor a blank slate — it is a stack with sharply uneven maturity across its own components, and the unevenness itself is the planning-relevant fact. Attention (FA4, cuDNN) and dense-GEMM autotuning (the CuTe-DSL Inductor backend) are close to production-grade for BF16, with FP8 forward workable and FP8 backward genuinely absent. MoE grouped GEMM — the kernel this design depends on most, given 200B active parameters spread thin across many experts — is the least mature component in the entire stack, with a documented order-of-magnitude gap between what ships by default and what the best-known technique achieves. That gap compounds with the routing/permutation overhead and sub-50%-of-layer-time GEMM share documented in the Megatron-Core MoE report: even a perfect grouped-GEMM kernel would still leave roughly half of MoE step time to communication and dispatch overhead (Ch18, Ch20) rather than FLOP-bearing compute.

Two decisions follow directly. First, budget real kernel-engineering headcount — not integration time — against the cooperative 2-CTA grouped-GEMM gap specifically; the 10x-to-15x public headroom documented above is large enough that closing even half of it changes the run's realized MFU more than any precision-format choice in Ch10. Second, do not plan the attention path around Blackwell-native block-scaled numerics (MXFP8/NVFP4 attention) landing before this run needs it — treat BF16 or per-tensor FP8 forward-only as the safe attention baseline, with a dedicated pilot (not a default assumption) for anything more aggressive, consistent with Ch10's own precision recommendations. The corrected 21.0% GB200 MXFP8-normalized MFU figure for DeepSeek-V3-scale MoE training (the underlying 1,048 TFLOPS/GPU measurement is MXFP8 per the report's own Table 11 — see above) is the most defensible real-world anchor available for what "kernel-stack-mature" currently looks like at this parameter-activity ratio (Ch18, Ch23) — a 10T/200B design should expect to start below that number on a novel architecture and treat closing the gap to it, and past it, as an explicit engineering program rather than an assumption the framework layer will absorb for free.
