# Chapter 15 — Kernels for Blackwell: FlashAttention-4, Grouped GEMM, and the Fusion Inventory

Every rooflines-and-MFU argument in this book (Ch16, Ch21) presumes a kernel layer that actually delivers the FLOPs a spec sheet promises. That layer turned over completely between Hopper and Blackwell — not incrementally, but architecturally: Blackwell moved matrix-multiply issuance from warp-scope to single-thread CTA-scope, introduced a new on-chip memory space that must be managed by hand, and shifted the attention kernel's binding constraint from tensor-core FLOPs to shared-memory bandwidth and exponential-unit throughput. As of July 2026 the software response to that shift is real, fast-moving, and unevenly finished: FlashAttention-4 exists and is fast, but its FP8 path is forward-only; DeepGEMM already speaks Blackwell's native microscaling format while attention kernels do not; and the single hardest unsolved problem in the entire stack — grouped GEMM at the small-M shapes a 10T-total/200B-active MoE actually produces — has a documented 10x gap between naive and hand-tuned code, with the best-known public technique still absent from every major library's default path. This chapter inventories that layer precisely enough to plan a build-or-fund list against it.

## The Blackwell Programming Model: tcgen05, TMEM, and 2-CTA MMA

The single biggest conceptual change for a kernel author moving from Hopper to Blackwell is where matrix-multiply issuance lives. Hopper's WGMMA instruction is warpgroup-scoped — CuTe's own layout algebra expresses this as `ThrID = Layout<_128>`, 128 cooperating threads jointly issuing one WGMMA — so a kernel author reasons about pieces of the MMA result living in each thread's registers. Blackwell's `tcgen05.mma` collapses that to `ThrID = Layout<_1>`: a single thread issues the instruction on behalf of the entire CTA. [SemiAnalysis, "Dissecting NVIDIA Blackwell Tensor Cores"](https://newsletter.semianalysis.com/) [CREDIBLE-HEARSAY] 2025–2026, corroborated by [NVIDIA CUTLASS tcgen05 programming docs](https://docs.nvidia.com/cutlass/) [OFFICIAL-DOCS] 2026. The practical consequence is that threads no longer implicitly own accumulator fragments in registers; the accumulator (and optionally operand A) now lives in **Tensor Memory (TMEM)**, a new 256KB-per-SM on-chip space wired directly to the tensor cores that software must explicitly allocate and manage — there is no implicit register-file fallback the way there was on Hopper and earlier. [NVIDIA CUTLASS tcgen05_programming.html](https://docs.nvidia.com/cutlass/) [OFFICIAL-DOCS] 2026

The second load-bearing primitive is **2-CTA (2-SM) cooperation**, CUTLASS's "cluster launch" mechanism: two adjacent CTAs in the same cluster jointly execute one `tcgen05.mma` via `cta_group::2` in PTX (`2CTA` in SASS), with only the leader CTA issuing the instruction while both CTAs' TMEM and SMEM remain addressable within the pair. Microbenchmarks show near-perfect weak scaling — doubling compute resources roughly doubles throughput — and for SMEM-bound shapes with N<128, splitting operand-B SMEM bandwidth demand across two SMs delivers **over 2x speedup**, not just 2x compute. [SemiAnalysis, "Dissecting NVIDIA Blackwell Tensor Cores"](https://newsletter.semianalysis.com/) [CREDIBLE-HEARSAY] 2025–2026 A companion primitive, **TMA multicast**, lets a single Tensor Memory Accelerator load push data into multiple CTAs' shared memory simultaneously via a hardware CTA bitmask, coalesced by the L2 Request Coalescer to approach the ideal `1/cluster_size` L2-bytes-per-SMEM-byte ratio. [same source] [CREDIBLE-HEARSAY] 2025–2026 And because clusters are guaranteed co-scheduled within a GPC, Blackwell adds **Cluster Launch Control (CLC)**: dynamic, persistent-kernel work-stealing across SMs, so an SM that finishes its assigned tile early can grab another SM's remaining work rather than idling. [SemiAnalysis; NVIDIA CUTLASS docs] [CREDIBLE-HEARSAY][OFFICIAL-DOCS] 2025–2026

CLC is worth flagging as more than a footnote. The identical "persistent kernel plus atomic-counter-driven work-stealing" pattern recurs independently in three unrelated places in this chapter — FA4's tile scheduler, the state-of-the-art NVFP4 grouped-GEMM worklog below, and Blackwell's own hardware-level CLC feature — which is convergent evidence that irregular-workload load balancing (varlen attention, small-M MoE) is becoming a first-class, hardware-assisted primitive on this generation of silicon rather than a software workaround layered on top of a fixed hardware model, the way Hopper-era kernels solved the same problem with clever static tile-size heuristics. [synthesis across primary sources cited below] [CREDIBLE-HEARSAY] 2026

One further architectural wrinkle matters for anyone assuming a single Blackwell code path: **consumer/workstation Blackwell (SM120/SM12x)** has a tensor-core programming model closer to Ampere's `mma.sync` than to datacenter SM100's `tcgen05`, forcing Triton, CUTLASS, FA4, and FlashMLA to each maintain three genuinely different code paths — Ampere-style, Hopper WGMMA, Blackwell tcgen05 — rather than one path with minor per-generation tweaks. A 10T/200B design living entirely on GB200/SM100 datacenter parts is insulated from this, but it explains why consumer-Blackwell kernel maturity (flagged below) lags so far behind datacenter Blackwell despite sharing an architecture name. ## FlashAttention-4: Redesigning Around Asymmetric Scaling

FlashAttention-4 (Zadouri, Hoehnerbach, Shah, Liu, Thakkar, Dao, "Algorithm and Kernel Pipelining Co-Design for Asymmetric Hardware Scaling") is not FA3 ported to Blackwell; it is a redesign forced by one hardware fact. Going from H100 to B200, BF16 tensor-core throughput scales 2.25x (1 to 2.25 PFLOPS dense on the HGX B200 SKU), while shared-memory bandwidth and the count of Special Function Units executing softmax's `exp` stay essentially flat. [arXiv:2603.05451](https://arxiv.org/abs/2603.05451) [PAPER] Mar 2026 The paper's own per-SM roofline (tile M=N=D=128) makes the shift concrete: in the forward pass, tensor cores need roughly 1,024 cycles and the exponential unit is *also* roughly 1,024 cycles — a genuine co-bottleneck — against 768 cycles of SMEM traffic. In the backward pass, which needs five MMAs instead of two, SMEM traffic balloons to roughly 3,328 cycles against 2,560 cycles of MMA compute, making shared-memory bandwidth the dominant backward-pass bottleneck by 25–60% depending on configuration. [arXiv:2603.05451](https://arxiv.org/abs/2603.05451) [PAPER] Mar 2026; roofline independently corroborated by [Colfax Research, "FlashAttention-4" technical summary](https://research.colfax-intl.com/) [BLOG] 2026

FA4's forward pipeline uses distinctly more warp roles than FA3's ping-pong scheme: a load warp driving TMA transfers, an MMA warp issuing async `tcgen05` matmuls, **two 4-warp softmax warpgroups** deliberately synchronized so they never compute exponentials simultaneously (spreading the SFU-plus-emulated-FMA exp load across time), a dedicated correction warpgroup applying rescaling off the critical path, and one to two epilogue warps — with one tile's MMA overlapping the adjacent tile's softmax. [Modal, "Reverse-Engineering FlashAttention-4"](https://modal.com/blog/) [CREDIBLE-HEARSAY] Sept 2025; confirmed by [arXiv:2603.05451](https://arxiv.org/abs/2603.05451) [PAPER] Mar 2026

Two specific micro-optimizations are worth naming because they are genuinely novel, not incremental. First, FA4 decomposes `2^x` via Cody-Waite range reduction and approximates the fractional part with a **degree-3 polynomial in Horner form** (coefficients `1.0 + 0.6951x + 0.2276x² + 0.0771x³`, fit via Sollya for minimal relative error), running this software path on the underused FMA datapath *in parallel with* the hardware `MUFU.EX2` instruction — splitting the softmax exponential workload across both rather than letting one displace the other. Second, **conditional softmax rescaling**: standard FlashAttention rescales the running output accumulator every time a new K-block potentially raises the row max; FA4 defers correction until the new max exceeds the old one by more than a tunable threshold τ (the paper cites τ ≈ log₂(256) = 8.0 in one configuration), reported to cut rescale operations by roughly **10x**. [arXiv:2603.05451](https://arxiv.org/abs/2603.05451) [PAPER] Mar 2026, the rescaling mechanism independently corroborated by [SemiAnalysis's live coverage of Tri Dao's Hot Chips 2025 talk](https://x.com/SemiAnalysis_) [CREDIBLE-HEARSAY] Aug 2025

The performance headline is consistent across primary and secondary sources: **1,605–1,613 TFLOPS/s BF16**, 71% of B200's ~2,250 TFLOPS BF16 dense peak (the HGX B200 SKU — see the SKU caveat below), 1.1–1.3x faster than cuDNN 9.13, and 2.1–2.7x faster than Triton, with the largest wins at 4K+ sequence lengths. [arXiv:2603.05451](https://arxiv.org/abs/2603.05451) [PAPER] Mar 2026 One conflation worth flagging for accuracy: a commonly repeated "~1.2 PFLOPS/s on B200" figure is actually FA3's FP8 number on H100 (arXiv:2407.08608, July 2024), not an FA4/B200 figure — FA4's own reported BF16 peak is ~1.6 PFLOPS, not 1.2. [arXiv:2407.08608](https://arxiv.org/abs/2407.08608) [PAPER] Jul 2024 vs [arXiv:2603.05451](https://arxiv.org/abs/2603.05451) [PAPER] Mar 2026

The "71% of B200's peak" figure itself carries the same SKU trap this book flags repeatedly (Ch8, Ch14, Ch16): the FA4 paper's denominator is the **HGX B200** dense BF16 peak of 2,250 TFLOPS, not GB200 NVL72's dense BF16 peak of 2,500 TFLOPS established as authoritative in this corpus's reconciliation. [notes/30-gb200-nvl72-peak-flops-reconciliation.md, this corpus] Against the GB200 NVL72 denominator, FA4's measured 1,605–1,613 TFLOPS/s would read as **64.2–64.5%** of peak rather than 71% — the kernel's absolute throughput is unchanged, but any reader porting "71%" into a GB200-denominated roofline calculation elsewhere in this book should use the corrected figure. Note that this specific percentage is this chapter's own arithmetic, not a figure stated in any note: the reconciliation note performs the analogous HGX-to-GB200 correction explicitly only for the Megatron-Core MFU figure below, and does not itself flag FA4's "71%" headline as SKU-ambiguous. The division (1,605–1,613 ÷ 2,500) is straightforward given both inputs are independently sourced, but it is a derived recomputation applied here by analogy, not a quoted "64.2–64.5%" from the corpus.

## FA4's Backward Pass and the July 2026 Repo-Verified Feature Matrix

Because papers lag shipped code, the state of FA4 as actually usable in a training stack is best established by reading the repository directly rather than the paper. A direct inspection of `Dao-AILab/flash-attention` (`flash_attn/cute/`, HEAD `0816ef1`, pulled 2026-07-13) against the March 2026 paper shows real, dated progress. [github.com/Dao-AILab/flash-attention](https://github.com/Dao-AILab/flash-attention) [CODE] verified 2026-07-13

**Backward pass**: fully present as of the paper, a 5-MMA chain (recompute S, compute dP, dV, dS, then dQ/dK) bottlenecked by SMEM bandwidth rather than compute, with 2-CTA mode as the main backward-specific optimization and a deterministic mode reaching **85–90% of nondeterministic throughput** via semaphore-locked, shortest-processing-time-first accumulation scheduling. [arXiv:2603.05451](https://arxiv.org/abs/2603.05451) [PAPER] Mar 2026 This is a genuinely fast maturation: Modal's engineers, reverse-engineering the kernel in September 2025, wrote that "a backwards version of this kernel is not available, but is expected imminently" — meaning FA4 went from no backward pass at all to a full, deterministic-capable backward in roughly five to six months. [Modal, "Reverse-Engineering FlashAttention-4"](https://modal.com/blog/) [CREDIBLE-HEARSAY] Sept 2025 vs [arXiv:2603.05451](https://arxiv.org/abs/2603.05451) [PAPER] Mar 2026

## cuDNN SDPA: Closing the Gap From Behind

cuDNN was first to ship *any* performant Blackwell attention kernel, ahead of both FA3-for-Blackwell (which never existed — WGMMA does not exist on SM100) and FA4, during a 2025 window the field describes as having "a lot of flux in attention implementations for new Blackwell hardware." [CREDIBLE-HEARSAY] 2026 By FA4's own account, NVIDIA and the FA4 authors collaborated starting with cuDNN 9.13/9.14 to fold FA4's techniques back into cuDNN, and "the latest cuDNN 9.19.1.2 now offers similar performance to FA4." [arXiv:2603.05451](https://arxiv.org/abs/2603.05451) [PAPER] Mar 2026 cuDNN's own 9.19-line release notes document a unified SDPA-forward backend descriptor, GPT-OSS-style attention-sink support for forward and backward on all SM80+ GPUs, and — the detail that should give a frontier-run planner pause — a fix for **MXFP8 SDPA-backward NaN/mismatch bugs specifically on Blackwell**. [NVIDIA cuDNN release notes](https://docs.nvidia.com/deeplearning/cudnn/) [OFFICIAL-DOCS] 2026

The net kernel-selection calculus for 2026: cuDNN and FA4 are close enough in raw BF16 dense/causal throughput (FA4 still leads by roughly 10–30% forward) that the practical decision is more about integration convenience — cuDNN ships in every NGC container, FA4 CuTe is a separate, fast-moving pip package on a weekly release cycle — than about a large performance gap. But the recency of the MXFP8 backward correctness fix implies that code path is young and under-exercised at production scale; treating "cuDNN supports MXFP8 attention" as a settled checkbox for a $2B run's precision recipe would be premature.

## Triton on Blackwell: Why Gluon Exists

Triton can *emit* `tcgen05` instructions — generic `tcgen05.cp` lowering and matrix descriptors landed by early 2026 — but multiple independent sources converge on the same conclusion: matching hand-tuned or CuTe-DSL-quality Blackwell attention codegen from Triton's high-level, automatically-pipelined abstractions failed. Blackwell's single-thread MMA issuance (with its leader-election requirement), multi-barrier pipelining, and TMEM allocation-with-contention-handling are exactly the low-level, hard-to-discover patterns a block-level compiler struggles to infer automatically. [patricktoulme.substack.com, "CuTile on Blackwell: NVIDIA's Compiler Moat Is Already Built"](https://patricktoulme.substack.com/) [CREDIBLE-HEARSAY] 2026

The Triton team's response was **Gluon**: a lower-level, Triton-adjacent frontend where the programmer manually allocates SMEM buffers, places barriers, and issues async-copy and `tcgen05`-MMA instructions explicitly — trading Triton's automatic pipelining for CuTe-DSL-like explicit control while staying inside the Triton/MLIR toolchain. Gluon tutorials, including a dedicated `06-tcgen05.py` walkthrough and a documented 2-CTA blocked-scaled-matmul example, live directly in the Triton repository, and a third-party worked example ("Inside a Blackwell 2-CTA GEMM: A Gluon Kernel Tour") demonstrates the pattern is reproducible outside NVIDIA and Dao-AILab, not an internal-only workaround. [triton-lang/triton, Gluon tutorials](https://github.com/triton-lang/triton) [CODE]; [tanmaypatil.com Blackwell 2-CTA Gluon GEMM post](https://tanmaypatil.com/) [BLOG] 2026 The severity here is worth stating plainly: this was not an incremental compiler improvement but an entirely new, lower-level frontend spun up specifically because high-level Triton abstractions could not close the gap — a stronger statement about tile-based-compiler limits on Blackwell than "Triton is a bit behind cuDNN."

GPU MODE's own lecture catalog corroborates this framing from the authors' side: Lecture 104, "Gluon: Tile-Based GPU Programming with Low-Level Control" (Peter Bell, Mario Lezcano, Keren Zhou), is the primary talk from Gluon's own authors. [github.com/gpu-mode/lectures](https://github.com/gpu-mode/lectures) [CODE] 2026 It is worth a direct correction here to a common assumption: GPU MODE's roughly 106-lecture catalog as of mid-2026 has almost no Blackwell/DeepGEMM-specific frontier-lab kernel disclosures — Tri Dao's FA4 talk was delivered at Hot Chips 2025, not GPU MODE, and no dedicated DeepGEMM lecture exists by title search. ## CuTe-DSL Adoption, ThunderKittens, and the Tile-Language Field

**CUTLASS 4.x's CuTe-DSL** — a Python authoring surface for the primitives above — is now the primary kernel-authoring layer for Blackwell at NVIDIA and Dao-AILab: FlashAttention-4 itself is written in CuTe-DSL rather than CUTLASS C++, DeepGEMM explicitly builds on CUTLASS/CuTe concepts, and Dao-AILab's **Quack** library reimplements RMSNorm, softmax, LayerNorm, and cross-entropy purely in CuTe-DSL, claiming 33–50% faster than `torch.compile` plus Liger on H100 at roughly 3TB/s memory throughput — near speed-of-light for memory-bound kernels. [NVIDIA CUTLASS docs](https://docs.nvidia.com/cutlass/) [OFFICIAL-DOCS] 2026

Stanford's **ThunderKittens**, an embedded-C++ tile-primitive DSL, shipped Blackwell support in March 2025 and a major 2.0 rewrite in February 2026 adding MXFP8/NVFP4 support, Cluster Launch Control scheduling, and explicit TMEM control; GEMM kernels are reported to "surpass or match cuBLAS" on B200 across BF16/MXFP8/NVFP4, with attention forward and backward running near cuDNN speed, "up to 2x faster than FA3 on H100" — note this last figure compares against FA3-on-H100, not FA4-on-B200. [Hazy Research blog, ThunderKittens 2.0](https://hazyresearch.stanford.edu/blog/) [OFFICIAL-BLOG] Feb 2026 The 2.0 release's engineering post-mortems are unusually concrete and worth citing for their specificity: removing unnecessary memory-fence instructions recovered **~20 TFLOP/s** in GEMM kernels; discovering that `tcgen05.cp` pipelines implicitly with `tcgen05.mma` with no explicit sync needed recovered **~500 TFLOP/s (~10%)** for NVFP4; and replacing manual leader-thread-selection loops with PTX's `elect.sync` instruction removed serialization and improved small-GEMM throughput by up to 10%. [Hazy Research blog] [OFFICIAL-BLOG] Feb 2026 These are exactly the kind of micro-optimizations invisible in a paper abstract that determine whether a hand-written kernel hits 80% or 95% of a competitor's throughput.

## Grouped GEMM at MoE Expert Shapes: The Small-M Problem

This is the single most consequential kernel gap for a 200B-active MoE design, and it is worth stating the mechanism precisely. A grouped GEMM for MoE shares K (hidden size) and N (expert intermediate size) across experts but gives each expert a different, data-dependent M — the number of tokens actually routed to it. At the expert-parallelism degree a 10T-total model needs, per-GPU per-expert M can be small, and small-M GEMMs are architecturally penalized on Blackwell specifically: TMEM occupancy caps at **one CTA per SM** once the accumulator plus scale-factor tensors fill the 512-column TMEM partition, so a naive per-expert-kernel-launch approach burns enormous time in launch and synchronization overhead relative to actual compute.

A detailed public worklog on NVFP4 grouped-GEMM optimization on B200 is unusually concrete about the resulting headroom. A reference/naive implementation runs at **238µs geomean** across four representative expert shapes; a hand-optimized version using a persistent kernel with a single flattened global work queue (atomic-counter-based tile scheduling across all experts, replacing per-expert launches), 4-stage warp-specialized pipelining (up from 2), TMA cluster multicast for operand-A reuse (up to 4x traffic reduction on memory-bound shapes), and problem-aware L2 cache-eviction hints reaches **23.8µs geomean — roughly 10x** — with individual shapes improving 7.4x to 13.9x. Even this optimized kernel only achieves **~50% average / ~80% steady-state tensor-core utilization** and **12.4% SM occupancy** (the TMEM-fits-one-CTA constraint again), and the author explicitly notes the best-known public submissions (~16µs geomean) go further via cooperative `cta_group::2` MMA — 2-CTA-shared TMEM, amortizing per-CTA overhead — and an M-major epilogue output layout enabling 256-bit vectorized stores, techniques the author's own kernel does not implement. [mufeezamjad.com, NVFP4 grouped-GEMM worklog](https://mufeezamjad.com/) [BLOG][CREDIBLE-HEARSAY] 2026 The competition-versus-production-library provenance of that ~16µs figure is itself unresolved — the source's language ("faster submissions") implies a competitive kernel-optimization context rather than a shipped default, meaning a training team may need to reimplement the technique rather than `pip install` it.

The takeaway for a 10T/200B design is unambiguous: off-the-shelf grouped GEMM at production MoE expert-parallelism degrees on Blackwell is closer to "a documented 10x of headroom exists between naive and hand-tuned" than "solved," and the state-of-the-art technique — 2-CTA cooperative MMA specifically for grouped GEMM — is not in any public library's default path as of mid-2026.

**DeepGEMM** (deepseek-ai/DeepGEMM, inspected directly) supports SM90 and SM100; SM90 grouped and dense GEMM is NT-layout only, while SM100 supports all four layouts (NT/TN/NN/TT) and requires LHS scaling factors in packed UE8M0 format — DeepGEMM already speaks Blackwell's native block-scaling format for FP8, the asymmetry with FA4 attention noted above. Two grouped-GEMM variants ship: a contiguous layout (tokens sorted by expert, M-axis grouped) for training-style large-batch MoE, and a masked layout for inference-decode-style variable-length CUDA-graph use. Cited peak is **1,550 TFLOPS on H800** (Apr 2025); a Blackwell-native peak figure for DeepGEMM itself was not found in this corpus's sources — a third-party guide's claim of "another 1.5–2x on top of H200" is not DeepGEMM's own published number and should be treated as an estimate, not a confirmed figure. The 2026-era changelog adds "Mega MoE" (overlapped NVLink communication fused with grouped GEMM — the mechanism behind DeepSeek-V4's wave-based EP overlap discussed in Ch4), an FP4 indexer kernel, and FP8×FP4 mixed-precision GEMM. [github.com/deepseek-ai/DeepGEMM](https://github.com/deepseek-ai/DeepGEMM) [CODE] verified 2026-07-13

**CUTLASS** ships dedicated grouped-GEMM examples (`75_blackwell_grouped_gemm`, `75_blackwell_grouped_gemm_block_scaled.cu`) claiming up to **5x throughput over H200-FP16** using Blackwell FP4. But this maturity is uneven across Blackwell SKUs in a way worth flagging for anyone tempted to treat "CUTLASS supports it" as a checkbox: consumer/workstation Blackwell (SM120/SM121, RTX Pro 6000, DGX Spark) has documented correctness bugs — a flashinfer-ai issue reports CUTLASS Grouped Block-Scaled GEMM producing invalid output on SM120 NVFP4 MoE, and a vLLM issue confirms CUTLASS 4.5 lacks a `CollectiveBuilder` specialization for tensor/token-scaled FP8 grouped GEMM on SM_120/SM_121 entirely. This is a distraction for a datacenter-only GB200 design (SM100 throughout), but it is evidence CUTLASS's grouped-GEMM code is newer and less battle-tested than its dense-GEMM code even in places it nominally "supports" a target. [github.com/flashinfer-ai/flashinfer issues #2723, #3628](https://github.com/flashinfer-ai/flashinfer) [CODE/ISSUE] 2026

**Transformer Engine**, per Megatron-Core's own 2026-Q2 roadmap, lists "GroupedGEMM and Gradient Accumulation Fusion," "GroupGEMM + SwiGLU + quantize fused MLP," and "grouped GEMM BF16/MXFP8 support with CUDA Graph compatibility" as completed items for the June 2026 (26.06) Megatron-Core release, alongside FP8/MXFP8 grouped-GEMM support and FP8 weights with BF16 optimizer states — confirming TE's grouped-GEMM plus fused-MLP path is production-integrated into the reference training framework as a pluggable backend alongside DeepGEMM. [Megatron-Core MoE user guide](https://docs.nvidia.com/megatron-core/) [OFFICIAL-DOCS] 2026 But even after this optimization, the Megatron-Core MoE technical report states that **routing and permutation overhead adds ~9% to layer execution time**, and — the more consequential figure — **GEMMs occupy under 50% of execution time for DeepSeek-V3-scale MoE training, versus roughly 70% for a dense Llama-3-405B run**: for current-generation MoE stacks, roughly half of step time is router, permute, communication, and other non-FLOP-bearing kernel overhead, not the matmuls themselves. ## FlashMLA, DSA, and the Varlen/THD Chain That Gates Packing

**FlashMLA** (deepseek-ai/FlashMLA) covers SM90 (H800/H100) dense BF16 decode, sparse FP8-KV decode, and sparse prefill, with SM100 (B200) adding dense MHA prefill on top of the SM90 feature set. Measured numbers: SM90 dense MLA decode up to 3,000 GB/s (memory-bound) or 660 TFLOPS (compute-bound); SM90 sparse MLA decode (FP8 KV) 410 TFLOPS; **B200 sparse MLA decode ~350 TFLOPS**, which the repository's own README candidly flags as "not really optimized yet"; B200 sparse MLA prefill up to 1,450 TFLOPS; B200 dense MHA prefill forward up to 1,460 TFLOPS, backward 1,000 TFLOPS. KV-cache quantization uses FP8 E4M3 with per-128-token scales, keeping the RoPE component unquantized in BF16. [github.com/deepseek-ai/FlashMLA](https://github.com/deepseek-ai/FlashMLA) [CODE] 2025–2026 A self-reported gap from the maintainers themselves is a rarer and more trustworthy kind of evidence than most vendor claims in this chapter.

Native Sparse Attention (NSA, the block-wise research design) and DeepSeek Sparse Attention (DSA, the shipped V3.2/V4 design using a finer-grained, token-wise "lightning indexer") are related but distinct, and DSA's kernels are genuinely open across two tiers: TileLang implementations (readable, research-oriented) released alongside DeepSeek-V3.2-Exp in September 2025, and production CUDA kernels split across DeepGEMM (indexer logit computation) and FlashMLA (the sparse-attention kernels themselves). A third-party TileLang reimplementation of the DSA lightning-indexer training operator exists at `lemyx/tilelang-dsa`, evidence of community reproduction beyond DeepSeek's own release. [vllm.ai/blog/2025-09-29-deepseek-v3-2](https://vllm.ai/blog/) [OFFICIAL-BLOG] Sept 2025; [github.com/lemyx/tilelang-dsa](https://github.com/lemyx/tilelang-dsa) [CODE] 2026 DeepSeek-V4 (April 2026, full architectural detail in Ch4) extends the indexer to 64 indexer heads, 4x key compression, and per-query top-k of 512–1024 tokens at 1M-token context.

**Varlen/THD support is the mechanism that makes sequence packing (Ch13) viable at all**, and it is worth stating why this chapter and not the dataloading chapter owns the kernel-level detail: packing only pays off if the attention and MoE kernels consuming packed sequences correctly track cumulative sequence-length offsets rather than materializing padded tensors. Transformer Engine's THD format (`[total_tokens, num_heads, head_dim]`) relies on `cu_seqlens` propagated through TE's fused attention kernels; TE v2.16 specifically added `pad_between_seqs` support for both non-context-parallel and context-parallel (ring/A2A and P2P variants) combined with FA3 plus THD attention, alongside FA4 support for head_dim=256. [NVIDIA TE docs] [OFFICIAL-DOCS] 2026 A known load-imbalance wrinkle recurs here identically to the small-M grouped-GEMM problem above: with varlen, SM tile assignments naturally have different mainloop lengths — a long sequence's tile does more work than a short sequence's tile — which is exactly the load-balance problem persistent, atomic-counter-based scheduling is designed to fix. That the same architectural answer (LPT/persistent scheduling) recurs across FA4's tile scheduler, small-M MoE grouped GEMM, and varlen attention is evidence this is becoming the general-purpose 2026 Blackwell kernel-engineering pattern, not a one-off trick specific to any single kernel.

On the multi-token-prediction and hybrid-attention side, FlexAttention's FA4 backend natively supports sliding-window as a built-in mask type alongside causal, ALiBi, document-mask, and soft-cap, with measured GB200 speedups over Triton of 1.4–2.1x forward and 1.8–2.2x backward for sliding-window patterns specifically, and up to 2.7x forward / 3x backward for document masks. [pytorch.org/blog/flexattention-flashattention-4](https://pytorch.org/blog/) [OFFICIAL-BLOG] 2026 Separately, Megatron-Core's 2026-Q2 roadmap lists "GDN kernel fusion on Blackwell" (marked "working") and "pre-gated delta rule kernel fusion" as in-progress items tied to Qwen3.5-style hybrid linear-plus-full-attention architectures (Ch5) — hybrid-attention kernel fusion is an acknowledged, actively-being-built gap, not a finished item, in the reference MoE training framework as of mid-2026. [github.com/NVIDIA/Megatron-LM issue #4815](https://github.com/NVIDIA/Megatron-LM/issues/4815) [OFFICIAL-DOCS/CODE] 2026

## The Fusion Inventory: Norm, RoPE, SwiGLU, Cross-Entropy, Optimizer

Below the attention and grouped-GEMM layer sits a set of smaller fusions whose aggregate effect on memory footprint and wall-clock is large enough to be load-bearing at frontier scale even though no individual fusion is architecturally novel.

**Norm/RoPE/SwiGLU fusion** is a solved, well-measured problem: aggregated fine-tuning-framework benchmarks report Triton-fused-kernel speedups of roughly **7x for RMSNorm, 5x for SwiGLU, and 2.3x for QK-RoPE fusion** relative to naive eager PyTorch, with the underlying pattern — normalizing `x + residual` without materializing the intermediate sum — standard across Liger-Kernel and Quack-style libraries. [CREDIBLE-HEARSAY] 2026 Fusing SwiGLU with FP8 amax/quantization specifically achieves latency "nearly identical" to standalone unquantized SwiGLU while producing FP8-quantized output directly — meaning the quantization step is effectively free when fused, which matters for any FP8-recipe pretraining stack (Ch8) where every activation needs a cast somewhere in the pipeline. [CREDIBLE-HEARSAY] 2026

**Chunked cross-entropy is the single highest-leverage, most completely solved fusion in this beat**, and it becomes non-optional rather than merely nice-to-have at the vocabulary sizes a 10T-total/200B-active multilingual/code design is likely to use. Apple's Cut Cross-Entropy computes only the correct-token logit plus an on-the-fly log-sum-exp over the vocabulary via a flash-memory-style kernel, never materializing the full [tokens × vocab] logit matrix; for Gemma-2 2B, this drops the loss computation's memory footprint from **24GB to 1MB** and total classifier-head training memory from **28GB to 1GB**, with no measured throughput or convergence regression. [arXiv:2411.09009](https://arxiv.org/abs/2411.09009) [PAPER] Nov 2024 Liger-Kernel's independent fused-linear-cross-entropy implementation reports roughly **5x memory reduction at vocab=163,840** via online-softmax chunked computation, and more broadly **40–55% peak-VRAM savings for vocab sizes above 100K**. The tradeoff is real, not free: the most memory-aggressive chunking configuration is reported to **more than double wall-clock latency** for the cross-entropy step specifically, since chunk count trades memory for latency directly. [Liger-Kernel docs](https://github.com/linkedin/Liger-Kernel) [PAPER/CODE] Oct 2024–2026 At frontier-scale vocabularies commonly exceeding 150–200K tokens, an unfused cross-entropy over a large batch-times-sequence-length would materialize a genuinely enormous logit tensor — this fusion is the difference between the loss computation fitting in memory at all, not a marginal optimization.

**Fused optimizer**: Apex's `FusedAdam` fuses the elementwise Adam update into one kernel and uses multi-tensor apply to batch the update across an entire model's parameter list into one or a handful of launches rather than one per tensor. [nvidia/apex fused_adam.py] [CODE] ongoing No evidence of a CuTe-DSL or `tcgen05`-specific rewrite of the fused-optimizer kernel was found in this corpus — this remains a straightforward memory-bound elementwise kernel where Hopper-era code likely runs unmodified on Blackwell, unlike the attention and grouped-GEMM kernels above, which needed ground-up redesign. [CREDIBLE-HEARSAY] 2026

## torch.compile Coverage and the Fusion Ceiling

TorchTitan (Ch17) uses regional compilation — `torch.compile` applied per-`TransformerBlock` rather than to the whole model graph — specifically to keep FSDP2/TP-compatible graphs free of graph breaks and to exploit the fact that structurally identical transformer layers only need compiling once. A compiled transformer block typically collapses from roughly 30 eager-mode kernel launches down to 5–8 fused kernels per layer: attention delegates to FA4 or cuDNN, matmuls delegate to cuBLAS or the CuTe-DSL Inductor backend with autotuned tile shapes, and layernorm collapses into a fused norm-residual op. [arXiv:2410.06511](https://arxiv.org/abs/2410.06511) [PAPER] 2024, still current practice per 2026 sources; [CREDIBLE-HEARSAY] 2026

What still needs hand-authored kernels in 2026, and is unlikely to stop needing them: attention (Inductor correctly delegates to FA4/cuDNN rather than attempting its own codegen, given the specialization complexity documented above), grouped GEMM and MoE dispatch-combine (no evidence Inductor auto-fuses these — Megatron-Core and TE ship them as explicit hand-written ops), and cross-entropy/CCE-style vocabulary chunking (the online log-sum-exp restructuring is an algorithmic change, not a mechanical fusion a pattern-matching compiler would discover). This is a stable, three-item boundary between what a 2026-era compiler can find automatically and what an infra team must still own by hand.

## The Gap List: What a 10T/200B Run Must Build or Fund

Every item below is a documented gap, not a speculative one — each has a primary-source or repo-verified basis cited above.

- **FA4 FP8/MXFP8 backward for attention.** FA4 CuTe's FP8 path is forward-only (`NotImplementedError` in the current repo); a training team wanting FP8 attention with a working backward must either wait for this to land, fall back to BF16 attention with FP8 elsewhere in the model, or fund the backward kernel directly. [github.com/Dao-AILab/flash-attention](https://github.com/Dao-AILab/flash-attention) [CODE] verified 2026-07-13
- **Blackwell-native block-scaled (MXFP8/NVFP4) attention on the training path.** No `mxfp8`/`nvfp4`/`block_scale` code exists anywhere in FA4 CuTe as of July 2026; cuDNN's MXFP8 SDPA-backward path has had recent Blackwell-specific correctness bugs. This is a bigger lift than FP8 backward alone — it needs a microscaling-aware softmax/rescaling redesign, not just a precision-format swap. [github.com/Dao-AILab/flash-attention](https://github.com/Dao-AILab/flash-attention) [CODE]; [NVIDIA cuDNN release notes](https://docs.nvidia.com/deeplearning/cudnn/) [OFFICIAL-DOCS] 2026
- **Production-grade cooperative 2-CTA MoE grouped GEMM in the public stack.** The 10x gap between naive and hand-tuned NVFP4 grouped GEMM at MoE expert shapes is documented; the technique closing the remaining gap to the best-known public results (2-CTA cooperative MMA, M-major epilogue) is in no major library's default path — this is the single highest-value kernel-engineering investment on this list given that GEMMs already occupy under 50% of MoE layer time even before accounting for this shape-efficiency loss. [mufeezamjad.com](https://mufeezamjad.com/) [BLOG]; [arXiv:2603.07685](https://arxiv.org/abs/2603.07685) [PAPER] Mar 2026
- **A non-Gluon, high-level-Triton path to competitive Blackwell attention performance**, if the stack has a reason to avoid CuTe-DSL/Gluon's lower-level programming burden — as of July 2026 this does not exist; Gluon's explicit-control model is the only Triton-toolchain path that works.
- **CuTe-DSL graduating out of public beta**, with a rigorous C++-parity benchmark suite — NVIDIA states "beta to GA by end of summer 2026" with no committed calendar date found in this corpus; a run committing to CuTe-DSL as production infrastructure before that graduation is committing to beta-quality tooling.
- **A Blackwell-native DeepGEMM peak-TFLOPS figure from DeepSeek itself.** The only DeepGEMM-published peak found is 1,550 TFLOPS on H800 (Apr 2025); every B200 figure in circulation is a third-party estimate, not a DeepSeek-published number — worth closing before betting a cost model on it.
- **Hybrid/linear-attention kernel fusion (GDN, pre-gated delta rule) on Blackwell**, currently in-progress rather than finished in Megatron-Core's own roadmap — relevant to the extent the architectural strawman in Ch5 adopts a hybrid-attention design.

## Implications for the 10T/200B Run

The kernel layer for this design is neither a solved commodity nor a blank slate — it is a stack with sharply uneven maturity across its own components, and the unevenness itself is the planning-relevant fact. Attention (FA4, cuDNN) and dense-GEMM autotuning (the CuTe-DSL Inductor backend) are close to production-grade for BF16, with FP8 forward workable and FP8 backward genuinely absent. MoE grouped GEMM — the kernel this design depends on most, given 200B active parameters spread thin across many experts — is the least mature component in the entire stack, with a documented order-of-magnitude gap between what ships by default and what the best-known technique achieves. That gap compounds with the routing/permutation overhead and sub-50%-of-layer-time GEMM share documented in the Megatron-Core MoE report: even a perfect grouped-GEMM kernel would still leave roughly half of MoE step time to communication and dispatch overhead (Ch16, Ch18) rather than FLOP-bearing compute.

Two decisions follow directly. First, budget real kernel-engineering headcount — not integration time — against the cooperative 2-CTA grouped-GEMM gap specifically; the 10x-to-15x public headroom documented above is large enough that closing even half of it changes the run's realized MFU more than any precision-format choice in Ch8. Second, do not plan the attention path around Blackwell-native block-scaled numerics (MXFP8/NVFP4 attention) landing before this run needs it — treat BF16 or per-tensor FP8 forward-only as the safe attention baseline, with a dedicated pilot (not a default assumption) for anything more aggressive, consistent with Ch8's own precision recommendations. The corrected 41.9% GB200 BF16 MFU figure for DeepSeek-V3-scale MoE training (itself conditional on the underlying 1,048 TFLOPS/GPU measurement being BF16 rather than FP8 — see the caveat above) is the most defensible real-world anchor this corpus provides for what "kernel-stack-mature" currently looks like at this parameter-activity ratio (Ch16, Ch21) — a 10T/200B design should expect to start below that number on a novel architecture and treat closing the gap to it, and past it, as an explicit engineering program rather than an assumption the framework layer will absorb for free.
