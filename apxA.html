Appendix A · The scaling cheat-sheet

Every formula you need, in one place

This is the dense reference card for the whole book: the roofline, the matmul critical batch, transformer FLOPs, the memory ledger, collective costs, per-strategy comms volume, and the inference rooflines — re-instantiated for the 10T-total / 200B-active run on 100,000 GB200 GPUs. Symbolic results are hardware-agnostic; you plug in your own C (peak FLOPs/s) and W (bandwidth).

~24 min read2 figuresgrounded in JAX How To Scale Your Model

This is a dense reference, not prose. Every formula below is drawn from the JAX How To Scale Your Model book (landing) and its referenced papers. Hardware-specific constants default to GB200/Blackwell where the book gives a Blackwell number; TPU v5e/v5p values are retained where they are the only anchor, since the book's symbolic results are hardware-agnostic and you re-instantiate the constants C (peak FLOPs/s) and W (bandwidth) for your silicon.

Runtime is bounded by max(compute, comms, memory). A 20% benchmark win is worthless at a 20% roofline-efficiency cost — every formula here exists to tell you which of the three terms you are paying.
Notation convention (used throughout)

B=batch in tokens (not sequences), T=query seq len, S=key/value seq len, D=d_model, F=d_ff, L=layers, N=query heads, K=KV heads, H=head dim, G=N/K (GQA group size), V=vocab, E=experts, k=experts/token, C=accelerator peak FLOPs/s, W=bidirectional link bandwidth (bytes/s), X/Y/Z=devices on a mesh axis, M_X/M_Y=number of mesh axes a strategy uses.

02The roofline (the master rule)

The roofline is the single most-used decision in the book: classify any op by its arithmetic intensity, compare against the accelerator's ridge point, and you know instantly whether you are compute- or bandwidth-bound.

data T = max(Tmath, Tcomms)   where   Tmath = FLOPs / C,   Tcomms = Bytes / W Good kernels overlap, so runtime is the max of the two floors. The pessimistic no-overlap bound is the sum Tmath + Tcomms.
QuantityFormulaUse it when
Compute timeT_math = FLOPs / Calways — the floor if perfectly compute-bound
Comms/mem timeT_comms = Bytes / Walways — the floor if perfectly bandwidth-bound
Runtime (overlap)T = max(T_math, T_comms)good kernels; the book's default estimate
Runtime (no overlap)T = T_math + T_commspessimistic upper bound
Arithmetic intensityI = FLOPs / Bytesclassify any op
Accelerator intensityI_acc = C / Wthe ridge point (240 v5e, 281 B200 HBM)

Achievable FLOP/stotal_FLOPs / max(T_math, T_comms)for roofline plots / MFU
 Rule: runtime is bounded by max(compute, comms, memory). A 20% benchmark win is worthless at a 20% roofline-efficiency cost.

03Matmul FLOPs, bytes, and the critical batch

For the canonical matmul X[B,D] · Y[D,F] → Z[B,F], the arithmetic intensity collapses to roughly the token batch when B ≪ D,F — which is why "critical batch" and "critical intensity" are the same number.

data I = B·D·F / (B·D + D·F + B·F) ≈ B   when   B ≪ D, F Intensity ≈ token batch. Compute-bound iff B > C/W_hbm ≈ 240 (v5e), 281 (B200), 295 (H100).
QuantityFormulaNote
FLOPs2·B·D·Ffactor 2 = multiply + add
Bytes (bf16)2BD + 2DF + 2BFload both inputs + write output
IntensityBDF/(BD+DF+BF) ≈ Bintensity ≈ token batch when B ≪ D,F
Compute-boundB > C/W_hbm≈240 (v5e), ≈281 (B200), ≈295 (H100)
Contraction FLOPs (general)2 × (product of all input dims, batch & contract counted once)no factor 2 for pure elementwise
Tiled-from-VMEM intensitybm·bn/(bm+bn); threshold ~10–20VMEM/SMEM is ~22× HBM BW
 The ridge-point batch is the threshold below which a matmul is memory-bound on HBM.

The precision levers move the critical batch by changing the bits-per-param vs bits-per-activation ratio. The general form is B_crit = (C/W_hbm) · (bits_per_param / bits_per_activation).

SchemeB_crit (v5e)Why
bf16 weights + bf16 FLOPs240baseline C/W_hbm
int8/fp8 weights, bf16 FLOPs120half the weight bytes, same FLOPs
int8 weights + int8 FLOPs240both peak ops/s and intensity double
per-example weight matrix (no reuse)≈2intensity is constant — always comms-bound, avoid in hot paths
 General: B_crit = (C/W_hbm) · (bits_per_param / bits_per_activation).
Sharded matmuls are almost always comms-bound on the contracting dim

For a sharded 2-chip matmul that shards the contracting D, you are compute-bound iff D/(2X) > C/W. For large models (D ≈ 8k) this needs X < 2, i.e. you are essentially always comms-bound on the contracting dim — prefer the all-reduce strategy. See Rooflines, Sharding.

jax — classify any matmul against the B200 ridge point
import jax.numpy as jnp

C       = 2.25e15            # B200 peak bf16 FLOP/s
W_hbm   = 8.0e12            # B200 HBM bytes/s
I_acc   = C / W_hbm         # ridge point ~= 281 FLOPs/byte

def is_compute_bound(B, D, F, dtype_bytes=2):
    flops  = 2 * B * D * F
    bytes_ = dtype_bytes * (B*D + D*F + B*F)
    I      = flops / bytes_                 # arithmetic intensity ~= B
    return I > I_acc, I, max(flops / C, bytes_ / W_hbm)

# e.g. B=4096, D=8192, F=32768 -> compute-bound (I ~= 4096 >> 281)

04Transformer FLOPs (training & inference)

The master rule, from Transformers, is the back-of-envelope you reach for constantly:

data Training FLOPs ≈ 6 · num_params · num_tokens  ·  Inference ≈ 2 · num_params · num_tokens 6 = 2 fwd + 4 bwd per matmul; inference is the 2 forward FLOPs only.

Per-layer FLOPs tell you which component dominates the budget (training figures are ×6 of the forward, inference ×2):

Attention fraction ≈ T/(8D)

With F=4D, D=NH, N=K, the attention fraction is ≈ T/(8D). Dot-product attention dominates training FLOPs only once T > 8D (~64K tokens at D=8k; ~37k at D=4608). Self-attn FLOPs = QKVO FLOPs at T = 2D. Causal masking halves useful attention FLOPs (needs a real kernel). SwiGLU's 3-matrix FFN keeps params/FLOPs fixed by setting d_ff = (2/3)· the ungated width (GLU Variants).

moe MoE compute-bound batch:   B > 120 · E/k  →  (E=256, k=8) ⇒ B > 3840 Params scale ×O(E), activated params/token ×k, sparsity E/k (8–64). Two all-to-alls (route + un-route) when expert-sharded, each ¼ of a single-axis all-gather.

05Memory: parameters, optimizer, KV cache, activations

The memory ledger has four entries: weights, optimizer state, KV cache, and activations. The first two set the training memory ceiling; the KV cache dominates inference; activations are rarely the bound at scale once you remat.

bf16 weights2inference / fwd

The max-model-per-device rule of thumb

Max model under pure DP+Adam = HBM_per_device / 10 (≈9B on v5p; ≈19B on a 192 GB B200). Under Muon (~6 B/param) the same device holds ~1.6× more parameters.

The KV cache is the memory term that scales with batch and sequence — it dominates inference memory (Transformers, Inference):

Paged KV cache kills fragmentation

OPT-13B FP16 = 800 kB/token, 1.6 GB at 2048 ctx (vLLM/PagedAttention). Block size 16 tokens; paging cuts the 62–80% fragmentation waste of Orca/FasterTransformer to ~0.

The memory toggle below shows how the four ledger entries trade off as you drop weight precision from BF16 to FP4 — the parameter and gradient bars shrink while optimizer state (fp32, dtype-independent) and activations stay fixed.

FlashAttention makes attention scratch O(N), not O(N²)

FlashAttention gives Θ(N²d²/M) HBM accesses (M=SRAM), block sizes B_c=⌈M/4d⌉, B_r=min(⌈M/4d⌉,d) (FlashAttention). It does more GFLOPs (75.2 vs 66.6) but 9× fewer HBM bytes (4.4 vs 40.3 GB) → 5.7× faster. Rabe-Staats lazy-softmax gives O(log n) / O(√n) memory on TPU (arXiv:2112.05682).

The latency floor — small buffers are latency-bound, not bandwidth-bound

T = max(T_min·|X|/2, V/W) with T_min ≈ 1 µs/hop. A buffer is latency-bound below W_unidir·T_min per device = 45 kB/device on v5e. AllGather hits ~95% of peak only at ~10 MB (~625 kB/device 16-way).

FSDP + TP combined is the workhorse at 100k-GPU scale, and the combination is what lets you stay compute-bound at a far smaller per-GPU batch:

data Compute-bound iff   B/N > α² / (M_X·M_Y·F) = 2550² / (2F) ≈ 100 tok/chip  (F=32768) This ~8× reduction (850 → ~100 tok/chip) vs pure FSDP is the reason to combine. Total comms time = max(12DF/(Y·W·M_X), 12BD/(X·W·M_Y)); optimal FSDP degree X_opt = √((B/F)·(M_X/M_Y)·N) = √(2BN/F).
Span nodes with model parallelism to divide the DP all-reduce

Model parallelism that spans nodes (PP/EP/TP) divides the DP all-reduce cost, letting you stay compute-bound at smaller per-GPU batch. Cross-pod/DCN: T_comms = 8DF/(M·W_dcn), M = chips/slice; compute-bound iff per-pod batch > C/W_dcn (= 73,440 tok/pod on v5p) (Training).

The DeepSeek-V3 mesh, for calibration

DeepSeek-V3 used 16-way PP × 64-way EP (8 nodes) × 2-way DP, no TP, at ~30k tok/GPU, NVLink ~160 GB/s / IB ~50 GB/s, all-to-all on 20 of 132 SMs, DualPipe hiding comms (DeepSeek-V3). 2D weight-stationary FFN sharding (Pope et al.): T_comm = 8BLE/(√n·BW), beats 1D when √n_chips > d_ff/d_model i.e. n_chips > 16 (ESTI).

Disaggregate prefill from decode, ~3:1

Speculative decoding wins whenever α > c

Expected tokens/step (1−α^(γ+1))/(1−α); walltime speedup (1−α^(γ+1))/((1−α)(γc+1)); wins whenever α > c (Leviathan et al.). Cuts HBM reads (memory-bound win), raises FLOPs/concurrency. Measured 2–3×; empirical acceptance ~3–4 tokens (NL), more for code.

10Worked back-of-envelope templates

Instantiate these for your run — they chain the formulas above into the three questions you actually ask: how long, how many chips to fit, how many to serve.

data Training time:   T = 6 · P · tokens / (n_chips · C · MFU) LLaMA-3 70B: 6·70e9·15e12 = 6.3e24 FLOPs; 8960-chip v5p pod @ 40% MFU → 44 days. On 4096 H100 @ 45% → ~40 days.
scaling_cheatsheet.pyback-of-envelope templatesillustrative
def training_days(P, tokens, n_chips, C, mfu):          # [1]
    flops = 6 * P * tokens                              # 6 = 2 fwd + 4 bwd
    return flops / (n_chips * C * mfu) / 86400

def min_chips_to_fit(P, hbm, opt_bytes=8, ckpt_bytes=0):  # [2]
    # weights(2) + optimizer(8 Adam / ~4 Muon) + checkpoints
    return ((2 + opt_bytes) * P + ckpt_bytes) / hbm

def serve_step_ms(P, kv_per_tok, B, S, W_hbm):         # [3]
    bytes_moved = P * 1 + kv_per_tok * S * B           # int8 weights
    return bytes_moved / W_hbm * 1e3                   # min step = load once

# LLaMA-3 70B @ 8960 v5p, 40% MFU, 15T tok:
training_days(70e9, 15e12, 8960, 4.59e14, 0.40)        # ~= 44 days  [4]

16·P·tokens is the whole budget. Divide by aggregate FLOP/s × MFU; /86400 converts seconds to days.

2Memory floor, not the real cluster. Muon drops opt_bytes 8→~4 (one fewer fp32 buffer). LLaMA-3 70B 4M-batch = 140 GB params + 560 GB optimizer + 20.9 TB checkpoints = 21.6 TB → ≥225 v5p.

3Decode is bandwidth-bound. Min step time = load weights + KV once: (B·KV + P_bytes)/W_hbm. LLaMA-3 70B int8 serve: 70 GB params + 41.9 GB KV (B=32, 8k) = 112 GB → 7 v5e → 112e9/(8·8.2e11) = 17 ms → 1882 tok/s.

4Use big clusters for FLOPs, not memory. The min-chips-to-fit figure is a floor; you provision far more for throughput. On 4096 H100 @ 45% the same run is ~40 days.

The 10T / 200B on 100k GB200 instantiation

Sparsity E/k sets activated-param FLOPs = 6·200e9·tokens; total params 10T sets memory = 10e12·(2 + optimizer)/sharding. With Muon (~6 B/param) the parameter+optimizer footprint is ~60 TB, spread by EP×FSDP across the 100k-GPU / ~1389-node NVL72 fabric. Keep EP within an NVL72 domain (72 GPUs) where AllToAll is 900 GB/s, since cross-node AllToAll degrades >4×; stay compute-bound by holding per-GPU batch above C/W ≈ 5625 for DP/FSDP and 2250e12/900e9 = 2500 for TP, capping TP at ~8-way and pushing the rest into PP+EP. MoE generation batch must exceed 120·E/k to keep experts compute-bound.
